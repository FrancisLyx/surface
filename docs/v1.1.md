# FastAPI 项目架构与开发规范 v1.1 增补内容

## 版本变更说明

本版本新增以下工程约束：

1. 明确依赖注入与对象生命周期；
2. 统一 Unit of Work 事务管理方式；
3. 阻止 ORM 对象跨越应用层边界；
4. 区分同步模块调用、进程内事件和可靠消息；
5. 引入 Transactional Outbox 处理关键事件；
6. 将启动失败、就绪检查和服务降级分级处理。

---

# 对原章节的修订

## 修订 4.2 Service 层职责

Service 负责：

* 业务规则和用例流程；
* 多个 Repository 的协作；
* 事务边界；
* 领域实体和值对象的创建；
* DTO 转换；
* 领域事件的产生；
* 非关键进程内事件或可靠事件的编排。

Service 禁止：

* 创建数据库 Session；
* 创建 Repository 的具体实现；
* 创建 Redis、HTTP Client、消息生产者等基础设施对象；
* 依赖 FastAPI 的 `Request`、`Response`、`HTTPException`；
* 返回仍然绑定数据库 Session 的 ORM 对象。

Service 可以创建纯业务对象：

```python
order = Order.create(...)
money = Money(amount=amount, currency=currency)
event = OrderCreatedEvent(order_id=order.id)
```

基础设施对象必须由依赖注入层统一组装：

```python
service = OrderService(
    unit_of_work=unit_of_work,
    event_bus=event_bus,
)
```

## 修订 5.1 跨模块调用

模块之间不得直接访问对方的：

* ORM Model；
* Repository；
* 数据库表；
* 内部工具函数；
* 非公开 Service 实现。

模块之间根据一致性需求选择通信方式。

| 场景          | 推荐方式                      |
| ----------- | ------------------------- |
| 查询另一个模块的数据  | 公开 Query Service 或 Facade |
| 需要立即返回结果    | 同步 Application Service    |
| 必须在同一事务中完成  | 同步调用或重新评估模块边界             |
| 非强一致副作用     | 进程内事件                     |
| 必须可靠执行的异步任务 | Outbox + 消息队列             |
| 跨服务长流程      | Saga 或补偿机制                |

允许：

```python
from app.modules.identity.public import IdentityQueryService
```

禁止：

```python
from app.modules.identity.models import User
from app.modules.identity.repository import UserRepository
```

事件不能代替所有同步调用。需要立即获得结果、校验业务条件或保证强一致性的操作，应使用明确的同步接口。

## 修订 6.4 数据库规范

增加以下规定：

1. 一个数据库 Session 只能服务于一个请求或一个独立任务；
2. 同一个 `AsyncSession` 不得在多个并发协程之间共享；
3. Repository 默认不得执行 `commit()`；
4. Service 决定业务事务边界；
5. Unit of Work 负责 Session 的提交、回滚和释放；
6. 项目必须在“显式提交”和“成功退出自动提交”之间选择一种策略；
7. 默认推荐显式提交；
8. ORM 到 DTO 的转换必须在 Session 有效期间完成；
9. 异步 ORM 查询必须避免不可控的隐式懒加载；
10. 测试环境应主动暴露 N+1 和懒加载问题。

关系加载策略：

* 多对一、小数据关系可考虑 `joinedload()`；
* 一对多集合通常优先考虑 `selectinload()`；
* 需要阻止意外查询时可使用 `raiseload()`；
* 禁止无依据地对所有关系统一使用 `joinedload()`。

## 修订 6.8 日志与可观测性

健康检查必须分为：

```text
/health/live
/health/ready
/health/startup
```

职责如下：

### `/health/live`

只判断进程是否仍在运行。

不应在该接口中访问：

* 数据库；
* Redis；
* 消息队列；
* 第三方服务。

外部依赖异常不应导致存活检查失败，否则可能引起实例反复重启。

### `/health/ready`

判断实例当前是否具备接收业务流量的条件。

可检查：

* 核心数据库；
* 必要缓存；
* 必要消息基础设施；
* 数据库 Schema 版本；
* 必须加载的运行时资源。

### `/health/startup`

判断应用启动初始化是否已经完成。

适用于：

* 模型加载；
* 大型缓存预热；
* 数据字典初始化；
* 较慢的启动流程。

# 11. 依赖注入与生命周期管理规范

## 11.1 Composition Root

项目必须设置明确的依赖组装入口，即 Composition Root。

推荐位置：

```text
app/
├── bootstrap.py
├── api/dependencies.py
└── core/lifespan.py
```

依赖组装代码负责：

* 创建 Repository；
* 创建 Unit of Work；
* 创建 Service；
* 注入配置；
* 注入外部 Client；
* 注入消息生产者；
* 控制对象生命周期。

Router 和 Service 不得承担依赖组装职责。

## 11.2 生命周期分类

依赖至少划分为三种生命周期。

### 应用级

整个应用进程共享：

* Settings；
* HTTP Client；
* Redis Client；
* 数据库 Engine；
* 消息队列连接池；
* 指标和追踪对象。

应用级依赖通常在 `lifespan` 中初始化和释放。

### 请求级

每次请求独立创建：

* SQLAlchemy Session；
* Unit of Work；
* Repository；
* 包含请求上下文的 Service；
* 当前用户上下文。

FastAPI 默认会在一次请求内缓存同一个依赖函数的返回值。一个请求中多次依赖同一个 Session Provider 时，应复用同一 Session。

### 瞬时对象

每次调用都可以重新创建：

* DTO；
* Command；
* 值对象；
* Mapper；
* 无状态领域服务。

## 11.3 配置和值提供者

静态配置优先通过类型化 Settings 注入：

```python
from typing import Annotated

from fastapi import Depends

from app.core.config import Settings, get_settings


SettingsDep = Annotated[Settings, Depends(get_settings)]
```

依赖工厂：

```python
def get_order_service(
    unit_of_work: UnitOfWorkDep,
    settings: SettingsDep,
) -> OrderService:
    return OrderService(
        unit_of_work=unit_of_work,
        run_mode=settings.run_mode,
    )
```

只有真正依赖当前应用实例或请求上下文时，才使用 `Request`：

```python
def get_request_id(request: Request) -> str:
    return request.state.request_id
```

禁止在 Service 中读取：

```python
request.app.state
os.environ
fastapi.Request
```

## 11.4 工厂函数的同步与异步选择

没有异步操作的依赖工厂使用普通 `def`：

```python
def get_order_repository(
    session: DbSessionDep,
) -> OrderRepository:
    return SqlAlchemyOrderRepository(session)
```

只有需要执行异步初始化时才使用 `async def`：

```python
async def get_external_context(
    client: ExternalClientDep,
) -> ExternalContext:
    return await client.load_context()
```

不要为了保持形式统一，将所有依赖工厂都声明为异步函数。

## 11.5 禁止事项

禁止在 Router 中执行：

```python
repo = SqlAlchemyOrderRepository(session)
service = OrderService(repo)
```

禁止在 Service 中执行：

```python
session = AsyncSession(...)
redis = Redis(...)
client = httpx.AsyncClient(...)
```

允许在 Service 中执行：

```python
order = Order.create(...)
event = OrderCreatedEvent(...)
dto = OrderMapper.to_dto(order)
```

# 12. Unit of Work 与事务边界规范

## 12.1 事务责任划分

事务职责必须明确：

```text
DI 层
负责创建和释放 UoW

Service 层
决定事务范围和提交时机

UoW
执行 commit、rollback、close

Repository
执行数据读写，不提交事务
```

Router 不得调用：

```python
session.commit()
session.rollback()
uow.commit()
```

## 12.2 默认采用显式提交

推荐的 Service 写法：

```python
class OrderService:
    def __init__(self, unit_of_work: UnitOfWork) -> None:
        self._uow = unit_of_work

    async def create_order(
        self,
        command: OrderCreate,
    ) -> OrderDTO:
        async with self._uow as uow:
            product = await uow.products.get_required(
                command.product_id
            )

            if product.stock < command.quantity:
                raise InsufficientStockError(product.id)

            product.reduce_stock(command.quantity)

            order = Order.create(
                product_id=product.id,
                quantity=command.quantity,
            )

            await uow.orders.add(order)

            dto = OrderMapper.to_dto(order)
            await uow.commit()

            return dto
```

显式提交的优点：

* 提交点清晰；
* 查询方法不会意外提交；
* 提前返回时行为明确；
* 代码审查时容易识别事务范围。

## 12.3 UoW 抽象接口

```python
from types import TracebackType
from typing import Protocol, Self


class UnitOfWork(Protocol):
    orders: OrderRepository
    products: ProductRepository
    outbox: OutboxRepository

    async def __aenter__(self) -> Self:
        ...

    async def __aexit__(
        self,
        exc_type: type[BaseException] | None,
        exc_value: BaseException | None,
        traceback: TracebackType | None,
    ) -> None:
        ...

    async def commit(self) -> None:
        ...

    async def rollback(self) -> None:
        ...
```

## 12.4 SQLAlchemy UoW 行为

当项目选择显式提交策略时，`__aexit__()` 必须遵循：

* 发生异常时回滚；
* 未显式提交时回滚；
* 无论成功或失败都关闭 Session；
* 不得在退出时隐式提交。

```python
class SqlAlchemyUnitOfWork:
    def __init__(
        self,
        session_factory: async_sessionmaker[AsyncSession],
    ) -> None:
        self._session_factory = session_factory
        self._session: AsyncSession | None = None
        self._committed = False

    async def __aenter__(self) -> "SqlAlchemyUnitOfWork":
        self._session = self._session_factory()

        self.orders = SqlAlchemyOrderRepository(self._session)
        self.products = SqlAlchemyProductRepository(self._session)
        self.outbox = SqlAlchemyOutboxRepository(self._session)

        return self

    async def commit(self) -> None:
        self._require_session()
        await self._session.commit()
        self._committed = True

    async def rollback(self) -> None:
        if self._session is not None:
            await self._session.rollback()

    async def __aexit__(
        self,
        exc_type: type[BaseException] | None,
        exc_value: BaseException | None,
        traceback: TracebackType | None,
    ) -> None:
        if self._session is None:
            return

        try:
            if exc_type is not None or not self._committed:
                await self._session.rollback()
        finally:
            await self._session.close()
            self._session = None

    def _require_session(self) -> None:
        if self._session is None:
            raise RuntimeError("Unit of Work is not active")
```

## 12.5 UoW 的使用边界

以下场景适合使用 UoW：

* 一个用例修改多个聚合；
* 一个业务操作写入多个 Repository；
* 数据变更和 Outbox 消息必须原子提交；
* 需要明确测试事务提交或回滚行为。

简单只读项目可以暂时使用请求级 Session，不必为了形式增加 UoW。

# 13. ORM 上下文隔离与 DTO 规范

## 13.1 ORM 泄漏红线

API 层不得接收仍然依赖 Session 的 ORM 对象。

禁止：

```python
async def get_order(...) -> Order:
    return await repository.get_by_id(order_id)
```

禁止直接将 ORM Model 作为接口响应：

```python
@router.get("/{order_id}")
async def get_order(...) -> Order:
    return await service.get_order(order_id)
```

主要风险：

* 序列化阶段触发懒加载；
* Session 已关闭后访问关联字段；
* 出现 `MissingGreenlet`；
* API 契约与数据库结构耦合；
* 无意暴露内部字段；
* 难以控制查询数量。

## 13.2 DTO 边界

推荐数据流：

```text
SQLAlchemy ORM Model
        ↓
Application Mapper
        ↓
Application DTO
        ↓
Router
        ↓
API Response Schema
```

Application DTO 不依赖 FastAPI：

```python
from dataclasses import dataclass
from decimal import Decimal
from uuid import UUID


@dataclass(frozen=True, slots=True)
class OrderDTO:
    id: UUID
    status: str
    total_amount: Decimal
    items: tuple["OrderItemDTO", ...]
```

Service 在 Session 有效期间完成转换：

```python
async def get_order_detail(
    self,
    order_id: UUID,
) -> OrderDTO:
    async with self._uow as uow:
        order = await uow.orders.get_with_items(order_id)
        return OrderMapper.to_dto(order)
```

Router 转换为 API Schema：

```python
@router.get("/{order_id}", response_model=OrderResponse)
async def get_order(
    order_id: UUID,
    service: OrderServiceDep,
) -> OrderResponse:
    dto = await service.get_order_detail(order_id)
    return OrderResponse.model_validate(dto)
```

小型项目允许 Service 直接返回 Pydantic DTO，但该 DTO 必须属于 Application 层，不能直接依赖 HTTP 状态码、Request 或 Response。

## 13.3 关联数据加载

Repository 必须显式声明查询所需的关联数据：

```python
statement = (
    select(Order)
    .options(selectinload(Order.items))
    .where(Order.id == order_id)
)
```

不得依赖序列化过程自动加载关联字段。

开发和测试阶段可对不允许懒加载的关系使用：

```python
lazy="raise"
```

或查询级：

```python
raiseload("*")
```

## 13.4 提交后的属性访问

异步 Session 推荐设置：

```python
async_sessionmaker(
    bind=engine,
    expire_on_commit=False,
)
```

但 `expire_on_commit=False` 不能代替 DTO 转换和显式加载。

它只能降低提交后访问基础属性时发生隐式查询的概率，不能保证所有关系都已经加载。

# 14. 模块解耦、事件总线与 Outbox

## 14.1 事件使用原则

事件适合表达已经发生的业务事实：

```text
OrderCreated
PaymentSucceeded
UserRegistered
InventoryReserved
```

事件名称使用过去式。

不推荐：

```text
CreateOrder
SendEmail
UpdatePoints
```

以上名称更接近 Command，而不是 Event。

## 14.2 进程内事件总线

进程内 Event Bus 适用于：

* 非关键日志；
* 本地缓存失效；
* 指标记录；
* 可容忍丢失的扩展逻辑；
* 单体应用内的轻量解耦。

接口示例：

```python
from typing import Protocol, TypeVar


EventT = TypeVar("EventT")


class EventBus(Protocol):
    async def publish(self, event: object) -> None:
        ...
```

Handler 注册应集中管理：

```python
event_bus.register(
    OrderCreatedEvent,
    update_order_metrics,
)
```

必须明确：

* Handler 串行还是并行；
* 一个 Handler 失败是否阻止其他 Handler；
* 是否向调用方传播异常；
* 是否允许重复处理；
* 是否记录处理结果。

## 14.3 关键业务禁止只使用进程内事件

以下业务不得只依赖进程内 Event Bus：

* 支付结果处理；
* 积分入账；
* 库存最终扣减；
* 财务记账；
* 退款；
* 必须发送的通知；
* 跨实例任务；
* 需要重试的任务。

进程退出、Worker 重启或实例崩溃都可能导致事件丢失。

## 14.4 禁止提交后直接发布关键事件

不可靠写法：

```python
await uow.commit()
await event_bus.publish(event)
```

该代码存在双写窗口：

```text
数据库提交成功
进程在消息发布前崩溃
事件永久丢失
```

关键事件必须使用 Transactional Outbox。

## 14.5 Transactional Outbox

业务数据和 Outbox 消息必须在同一个数据库事务中提交：

```python
async with self._uow as uow:
    order = Order.create(...)

    await uow.orders.add(order)
    await uow.outbox.add(
        OutboxMessage.from_event(
            OrderCreatedEvent(
                order_id=order.id,
                user_id=order.user_id,
            )
        )
    )

    dto = OrderMapper.to_dto(order)
    await uow.commit()

return dto
```

Outbox 表建议包含：

```text
id
event_type
aggregate_type
aggregate_id
payload
status
created_at
published_at
retry_count
next_retry_at
last_error
```

独立 Worker 负责：

1. 查询待发送消息；
2. 发布到消息队列；
3. 标记为已发布；
4. 失败时记录原因并重试；
5. 超过阈值后进入死信处理。

## 14.6 消费者幂等性

消息系统通常只能可靠实现“至少一次”投递，因此消费者必须支持重复消息。

推荐建立 Inbox 或消费记录表：

```text
consumer_name
message_id
processed_at
```

处理流程：

```text
检查 message_id 是否已消费
    ↓
未消费则执行本地事务
    ↓
记录消费结果
    ↓
提交事务
```

业务唯一约束也应作为最终防线。

## 14.7 事件契约管理

事件结构一旦被其他模块或服务消费，就属于公开契约。

要求：

* 事件包含唯一 `event_id`；
* 包含 `occurred_at`；
* 包含 Schema 版本；
* 新增字段优先保持向后兼容；
* 禁止直接传输 ORM 对象；
* Payload 使用可序列化的基础类型；
* 事件消费者不得依赖生产模块的内部 Model。

示例：

```python
class OrderCreatedEvent(BaseModel):
    event_id: UUID
    schema_version: int = 1
    occurred_at: datetime
    order_id: UUID
    user_id: UUID
    total_amount: Decimal
```

# 15. 启动阶段 Fail-Fast 与降级策略

## 15.1 分级原则

不是所有依赖异常都应阻止进程启动。

故障必须分为三类：

### A 类：程序不变量错误

必须阻止启动：

* 必填配置缺失；
* Secret 格式错误；
* DSN 格式无效；
* 不支持的运行模式；
* 数据库 Schema 版本不兼容；
* 必须加载的本地资源不存在；
* 核心依赖对象无法构造；
* 安全配置明显无效。

### B 类：核心基础设施暂时不可用

通常不必反复终止进程，但 readiness 必须失败：

* 数据库暂时不可访问；
* Redis 暂时不可访问且业务强依赖 Redis；
* 必需消息基础设施暂时不可访问；
* 关键远程配置服务暂时不可用。

应用进程可以保持运行并持续恢复连接，但在恢复前不得接收业务流量。

### C 类：非核心依赖不可用

应用可以降级运行：

* 邮件服务；
* 短信服务；
* 非必要推荐服务；
* 非关键分析平台；
* 可选第三方接口。

系统应：

* 记录错误；
* 触发告警；
* 返回明确的降级结果；
* 使用重试、Outbox 或任务队列恢复；
* 不得让整个应用启动失败。

## 15.2 Lifespan 职责

`lifespan` 负责：

* 校验关键配置；
* 初始化连接池和共享 Client；
* 注册依赖；
* 启动必要后台组件；
* 释放资源。

示例：

```python
from collections.abc import AsyncIterator
from contextlib import asynccontextmanager

from fastapi import FastAPI


@asynccontextmanager
async def app_lifespan(
    app: FastAPI,
) -> AsyncIterator[None]:
    settings = load_settings()
    validate_critical_settings(settings)

    database = create_database(settings.database_url)
    redis = create_redis_client(settings.redis_url)
    http_client = create_http_client()

    app.state.settings = settings
    app.state.database = database
    app.state.redis = redis
    app.state.http_client = http_client

    await verify_schema_compatibility(database)

    try:
        yield
    finally:
        await http_client.aclose()
        await redis.aclose()
        await database.dispose()
```

## 15.3 外部依赖检查限制

启动检查必须设置：

* 超时时间；
* 最大重试次数；
* 指数退避；
* 明确日志；
* 不记录 Secret；
* 不执行昂贵查询。

禁止：

```python
await third_party_api.perform_real_business_request()
```

推荐：

```python
await asyncio.wait_for(
    database.check_connection(),
    timeout=3,
)
```

## 15.4 Readiness 状态

应用应维护可观察的就绪状态：

```python
@dataclass
class ReadinessState:
    database_ready: bool = False
    redis_ready: bool = False
    messaging_ready: bool = False

    @property
    def ready(self) -> bool:
        return (
            self.database_ready
            and self.redis_ready
            and self.messaging_ready
        )
```

实际项目应根据依赖的重要等级决定哪些字段影响 `ready`。

## 15.5 降级策略

每个外部依赖必须在设计阶段定义失败行为：

| 依赖       | 失败策略                |
| -------- | ------------------- |
| 主数据库     | readiness 失败，拒绝业务流量 |
| Redis 缓存 | 回源数据库或关闭缓存          |
| 消息队列     | 写入 Outbox，等待恢复      |
| 邮件服务     | 任务重试，不阻塞主流程         |
| 支付服务     | 返回明确失败，不假定支付成功      |
| 推荐服务     | 返回默认结果              |
| 日志平台     | 本地标准输出兜底            |

禁止捕获异常后静默忽略：

```python
try:
    await client.send(...)
except Exception:
    pass
```

必须记录异常并执行明确的恢复或降级策略。

# 16. 架构复杂度控制

新增 UoW、DTO、Event Bus 和 Outbox 后，团队必须防止过度抽象。

## 16.1 可以暂缓的设计

以下项目可以暂时不引入 UoW：

* 只读 API；
* 单表 CRUD；
* 没有跨 Repository 事务；
* 没有可靠事件需求。

以下项目可以暂时不引入 Event Bus：

* 只有一个业务模块；
* 模块间调用关系简单；
* 所有操作都要求同步返回；
* 团队尚未建立事件监控能力。

以下项目可以暂时不引入 Outbox：

* 事件允许丢失；
* 事件仅用于本地指标；
* 事件不改变关键业务状态。

## 16.2 必须引入的信号

出现以下情况时，应引入 UoW：

* 一个用例修改多张表；
* 多个 Repository 必须原子提交；
* 事务提交和回滚逻辑开始重复；
* 需要同时保存业务数据和 Outbox。

出现以下情况时，应引入 Outbox：

* 数据库成功但消息丢失会造成状态不一致；
* 消息必须重试；
* 服务运行多个实例；
* 事件驱动积分、库存、支付或通知；
* 需要追踪消息发送状态。

## 16.3 最终默认方案

中型业务系统推荐采用：

```text
FastAPI 原生 Depends
+ 明确 Composition Root
+ 请求级 AsyncSession
+ 显式提交 UoW
+ ORM 到 DTO 隔离
+ 同步公开模块接口
+ 非关键进程内事件
+ 关键事件 Transactional Outbox
+ 分级 Fail-Fast
+ Liveness / Readiness / Startup 分离
```

该方案在工程严谨度、测试便利性和实现复杂度之间保持平衡，避免把所有问题都交给事件、抽象接口或基础设施解决。
