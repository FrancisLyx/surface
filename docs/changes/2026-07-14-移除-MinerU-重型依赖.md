# chore: 移除 MinerU 重型依赖

## 需求理解
根据反馈，`mineru[all]` 依赖体积过大，需要先从项目依赖中移除，并同步清理锁文件中由它引入的重型依赖链。

## 改动文件列表
- `pyproject.toml`：删除 `mineru[all]>=3.4.4` 依赖声明。
- `uv.lock`：通过 `uv lock` 重新解析依赖，移除 MinerU、Torch、TorchVision、Accelerate、HuggingFace 以及 CUDA/模型推理相关传递依赖。

## 关键实现说明
- 直接从项目主依赖列表移除 `mineru[all]`，避免后续安装时继续拉取大体积机器学习和模型推理依赖。
- 使用 `uv lock` 让锁文件与当前依赖声明一致，删除仅由 MinerU 依赖链引入的包。
- 已执行 `uv sync`，当前 `.venv` 也已卸载相关包，避免本地环境残留。

## 风险与兼容性影响
- 如果后续功能依赖 MinerU 做 PDF 或文档解析，需要改为按需引入更轻量依赖或单独设计可选依赖组。
- 锁文件大幅删减，CI 或部署环境下一次同步会移除这些包，需确认没有隐藏代码路径直接 import MinerU/Torch 相关模块。
- 当前策略分析和 AkShare 相关功能不依赖 MinerU，预期不受影响。

## 验证步骤
- `uv lock`：解析成功，锁文件收敛到 104 个包。
- `uv sync`：成功卸载 MinerU 相关依赖链中的 85 个已安装包。
- `rg -n "mineru|torch|torchvision|accelerate|huggingface" pyproject.toml uv.lock`：无匹配。
- `uv pip list | rg -i "mineru|torch|torchvision|accelerate|huggingface"`：无匹配。
- `uv run pytest tests/test_strategy_routes.py tests/test_akshare_client.py`：17 passed，1 个 passlib/crypt Python 3.13 弃用警告。

## 前端测试建议
不涉及前端变更。

## 代码审查建议
- 重点确认 `uv.lock` 中删除的依赖均来自 MinerU 传递依赖链，不影响当前后端运行时必要依赖。
- 检查项目中是否存在未被测试覆盖的 MinerU/Torch/HuggingFace import 或文档解析入口。
- 关注后续如需文档解析能力时，应通过可选依赖或服务隔离方式引入，避免主应用安装体积膨胀。

## 回滚建议
- 如确认必须恢复 MinerU 能力，可重新添加 `mineru[all]>=3.4.4` 到 `pyproject.toml` 并执行 `uv lock && uv sync`。
- 如只需要部分文档解析能力，优先新增更小范围依赖，而不是恢复 `mineru[all]`。
