# FastAPI 项目开发手册 v1.0

> 配套《FastAPI 项目架构与开发规范》（完整版 v1.2，文件 `架构与开发规范.md`）。规范讲"为什么这样约束"，本手册讲"具体怎么落地写代码"。
> 二者冲突时以架构规范为准；本手册中的 `见规范 §X` 指向该规范对应章节（章节号一致）。

---

## 0. 关于本手册

### 0.1 定位

* 面向在本架构下写业务代码的开发者，提供从环境搭建到提交上线的完整操作路径。
* 通用手册，不绑定具体业务；示例统一使用中性领域 `Order` / `Product`。
* 所有代码示例基于规范的默认技术选型：异步 SQLAlchemy + 用例级工厂 UoW + DTO 隔离 + 领域异常边界映射。

### 0.2 术语速查

| 术语 | 含义 | 规范出处 |
| --- | --- | --- |
| Composition Root | 依赖统一组装入口 | 规范 §11.1 |
| UoW | Unit of Work，事务一致性边界 | 规范 §12 |
| DTO | 应用层数据传输对象，不依赖 ORM/HTTP | 规范 §13 |
| Outbox | 事务性发件箱，保证消息可靠投递 | 规范 §14.5 |
| DomainError | 领域异常基类，由边界层映射为 HTTP | 规范 §17 |

## 1. 环境与工具链

### 1.1 依赖管理

* 使用单一锁文件管理依赖（`uv.lock` / `poetry.lock`），提交入库，禁止手改。
* 依赖分组：运行时依赖与开发依赖（`ruff`、`mypy`、`pytest` 等）分开声明。
* 新增依赖前先问"标准库能否解决"（奥卡姆剃刀）；引入后在 PR 说明理由。

```bash
uv sync                 # 按锁文件安装
uv add sqlalchemy       # 新增运行时依赖
uv add --dev pytest     # 新增开发依赖
```

### 1.2 配置

* 所有配置走类型化 `Settings`，禁止在业务代码里读 `os.environ`（见规范 §11.3）。
* 本地用 `.env`，`.env` 不入库；提供 `.env.example` 作为字段清单。
* 密钥只从环境注入，日志与报错中禁止打印密钥。

### 1.3 本地运行

```bash
uv run uvicorn app.main:app --reload      # 开发热重载
uv run alembic upgrade head               # 应用数据库迁移
uv run pytest                             # 全量测试
```

### 1.4 代码质量工具

统一在 `pre-commit` 中挂载，提交即校验：

| 工具 | 职责 |
| --- | --- |
| ruff format | 代码格式化 |
| ruff check | Lint、导入排序 |
| mypy | 静态类型检查 |
| pytest | 单元/集成测试 |

> 编译/类型/测试报错必须修复，禁止用简化或降级绕过（见全局规范）。

## 2. 工程流程

### 2.1 分支模型

* `main`：可发布分支，受保护，只接受通过评审与 CI 的 PR。
* 特性分支：`feat/<模块>-<简述>`；修复：`fix/<简述>`；文档：`docs/<简述>`。
* 分支生命周期短，尽量小步提交、频繁合并，避免长期分叉。

### 2.2 提交规范

采用 Conventional Commits：

```text
<type>(<scope>): <subject>

type: feat | fix | refactor | docs | test | chore | perf
scope: 模块名，如 order / identity
```

* 一次提交只做一件事，提交信息说清"改了什么、为什么"。
* 禁止把格式化噪声与逻辑改动混在同一提交。

### 2.3 PR 与评审

提交 PR 前作者自查（对照 §7 清单），评审重点：

1. **分层是否正确**：Service 是否越界创建基础设施 / 返回 ORM（见规范 §4.2、§13.1）。
2. **事务边界是否清晰**：提交点是否唯一、提前返回是否安全（见规范 §12.2）。
3. **ORM 是否泄漏到 API**：响应是否为 DTO/Schema 而非 ORM Model。
4. **异常是否走边界映射**：Service 是否抛领域异常而非 `HTTPException`（见规范 §17）。
5. **可靠性**：关键事件是否走 Outbox；消费者是否幂等（见规范 §14）。
6. **测试**：是否覆盖提交/回滚、异常路径、幂等。

### 2.4 CI 门禁

CI 必须全绿才能合并：`ruff format --check` → `ruff check` → `mypy` → `pytest`（含集成测试）。

### 2.5 迁移与发布顺序

数据库迁移遵循"先兼容、后清理"，避免发布过程中新旧代码与新旧 Schema 不匹配：

```text
1. 加列/加表（可空、有默认）—— 向后兼容
2. 部署新代码（同时兼容新旧结构）
3. 数据回填
4. 收紧约束 / 删除旧列 —— 下一个发布周期
```

* 迁移与代码同一 PR 提交，保证可回溯。
* 回滚代码时，已应用的向后兼容迁移无需回滚；破坏性迁移禁止在同一发布内进行。

## 3. 项目结构与依赖方向

### 3.1 目录布局

```text
app/
├── main.py                      # 应用入口，装配 router / 中间件 / lifespan
├── bootstrap.py                 # Composition Root：组装依赖
├── core/
│   ├── config.py                # 类型化 Settings
│   ├── lifespan.py              # 启动/关闭生命周期
│   ├── exceptions.py            # DomainError 层次与边界映射
│   └── logging.py
├── api/
│   ├── dependencies.py          # FastAPI Depends 工厂
│   └── routes/<module>/
│       ├── <module>_view.py     # Router，仅编排
│       └── <module>_schema.py   # API 请求/响应 Schema（Pydantic）
├── services/
│   └── <module>_service.py      # 用例编排 + 事务边界
├── domain/
│   └── <module>/                # 领域实体、值对象、领域异常
├── repositories/
│   └── <module>_repository.py   # 数据读写，不提交事务
├── db/
│   ├── models/                  # SQLAlchemy ORM Model
│   ├── uow.py                   # UnitOfWork 实现
│   └── session.py               # engine / async_sessionmaker
├── dtos/
│   └── <module>_dto.py          # 应用层 DTO + Mapper
├── events/
│   ├── <module>_events.py       # 领域事件定义
│   └── bus.py                   # 进程内事件总线
└── outbox/
    ├── model.py                 # Outbox 表
    └── worker.py                # 发件箱 Worker（独立进程）
```

> 模块化组织时可按 `app/modules/<module>/{api,service,domain,repository,...}` 收拢，同一模块内聚。跨模块只通过公开接口访问（见规范 §5.1）。

### 3.2 依赖方向（只允许向内依赖）

```text
API (Router/Schema)
      ↓
Service（用例）
      ↓
Domain（实体/值对象/领域异常）   ← 不依赖任何外层
      ↑
Repository / UoW / DTO Mapper（实现细节，被 Service 调用）
```

硬规则：

* Domain 不 import 任何框架、ORM、HTTP。
* Service 不 import `fastapi`、不创建 Session/Client（见规范 §11.5）。
* Router 不写业务逻辑、不组装依赖、不碰事务。

## 4. 编码规范与命名

### 4.1 命名约定

| 对象 | 约定 | 示例 |
| --- | --- | --- |
| 模块/包 | 小写下划线 | `order`, `fund_favorite` |
| 类 | 大驼峰 | `OrderService`, `OrderCreatedEvent` |
| 函数/变量 | 小写下划线 | `create_order`, `unit_of_work` |
| 常量 | 全大写 | `MAX_RETRY` |
| ORM Model | 名词单数 | `Order`, `Product` |
| Repository | `<聚合>Repository` | `OrderRepository` |
| Service | `<模块>Service` | `OrderService` |
| DTO | `<名>DTO` | `OrderDTO` |
| API Schema | 请求 `<名>Request` / 响应 `<名>Response` | `OrderCreateRequest` |
| 领域事件 | 过去式 `<名>Event` | `OrderCreatedEvent` |
| 命令 | 祈使 `<名>Command` | `CreateOrderCommand` |
| 领域异常 | `<名>Error` | `InsufficientStockError` |
| 依赖工厂 | `get_<名>` | `get_order_service` |

* 事件名用过去式（已发生的事实），不要用 `CreateOrder` 这种命令式（见规范 §14.1）。
* DTO 与 API Schema 分开命名，不复用同一个类跨层。

### 4.2 类型注解

* 所有公开函数带完整类型注解；用现代语法 `X | None`、`list[X]`、`dict[K, V]`。
* Repository/UoW 接口用 `Protocol` 定义（见规范 §12.3），便于测试替身。

### 4.3 导入顺序

标准库 → 第三方 → 本项目，三组之间空行；由 ruff 自动排序，不手动调整。

### 4.4 同步与异步

* 依赖工厂无异步操作用普通 `def`，仅需 `await` 时才 `async def`（见规范 §11.4）。
* 不要为了"统一形式"把无异步逻辑的工厂写成 `async`。

### 4.5 异常

* 领域/Service 只抛 `DomainError` 子类，不抛 `HTTPException`。
* 禁止 `except Exception: pass` 静默吞异常（见规范 §15.5）；必须记录并执行明确恢复/降级。

### 4.6 日志

* 结构化日志，带 `request_id` / `correlation_id`。
* 禁止打印密钥、完整 Token、用户敏感信息。

## 5. 分层开发流程：新增一个用例（端到端）

以"创建订单"为例，从上到下依次落地。**建议按 5.1 → 5.8 的顺序写，每步只关注单层职责。**

### 5.1 定义 API Schema（`api/routes/order/order_schema.py`）

```python
from decimal import Decimal
from uuid import UUID

from pydantic import BaseModel, Field


class OrderCreateRequest(BaseModel):
    product_id: UUID
    quantity: int = Field(gt=0)


class OrderResponse(BaseModel):
    id: UUID
    status: str
    total_amount: Decimal
```

### 5.2 定义应用层 DTO 与 Mapper（`dtos/order_dto.py`）

```python
from dataclasses import dataclass
from decimal import Decimal
from uuid import UUID


@dataclass(frozen=True, slots=True)
class OrderDTO:
    id: UUID
    status: str
    total_amount: Decimal


class OrderMapper:
    @staticmethod
    def to_dto(order: "Order") -> OrderDTO:
        return OrderDTO(
            id=order.id,
            status=order.status,
            total_amount=order.total_amount,
        )
```

DTO 不依赖 FastAPI，转换必须在 Session 有效期内完成（见规范 §13.2）。

### 5.3 领域模型与领域异常（`domain/order/`）

```python
class InsufficientStockError(ConflictError):
    def __init__(self, product_id: UUID) -> None:
        self.product_id = product_id
        super().__init__(f"insufficient stock: {product_id}")


class Order:
    @classmethod
    def create(cls, product_id: UUID, quantity: int) -> "Order":
        ...  # 纯业务对象，无 Session、无 IO
```

### 5.4 Repository（`repositories/order_repository.py`）

```python
from typing import Protocol
from uuid import UUID

from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy.orm import selectinload


class OrderRepository(Protocol):
    async def add(self, order: Order) -> None: ...
    async def get_with_items(self, order_id: UUID) -> Order: ...


class SqlAlchemyOrderRepository:
    def __init__(self, session: AsyncSession) -> None:
        self._session = session

    async def add(self, order: Order) -> None:
        self._session.add(order)

    async def get_with_items(self, order_id: UUID) -> Order:
        stmt = (
            select(Order)
            .options(selectinload(Order.items))   # 显式声明关联加载
            .where(Order.id == order_id)
        )
        order = await self._session.scalar(stmt)
        if order is None:
            raise NotFoundError(order_id)
        return order
```

Repository 只读写，**默认不 `commit()`**（见规范 §6.4）。

### 5.5 Service（`services/order_service.py`）

```python
from collections.abc import Callable


class OrderService:
    def __init__(self, uow_factory: Callable[[], UnitOfWork]) -> None:
        self._uow_factory = uow_factory

    async def create_order(self, command: CreateOrderCommand) -> OrderDTO:
        async with self._uow_factory() as uow:
            product = await uow.products.get_required(command.product_id)

            if product.stock < command.quantity:
                raise InsufficientStockError(product.id)

            product.reduce_stock(command.quantity)
            order = Order.create(product.id, command.quantity)
            await uow.orders.add(order)

            # 关键事件与业务数据同事务写入 Outbox（见 §6.5）
            await uow.outbox.add(OutboxMessage.from_event(
                OrderCreatedEvent.of(order),
            ))

            dto = OrderMapper.to_dto(order)   # Session 有效期内转 DTO
            await uow.commit()                # 唯一提交点
            return dto
```

Service 注入 **UoW 工厂**，每次 `async with` 一个用例事务（见规范 §12.2）。

### 5.6 依赖装配（`api/dependencies.py`）

```python
from typing import Annotated
from fastapi import Depends


def get_uow_factory(
    session_factory: SessionFactoryDep,
) -> Callable[[], UnitOfWork]:
    return lambda: SqlAlchemyUnitOfWork(session_factory)


def get_order_service(
    uow_factory: Annotated[..., Depends(get_uow_factory)],
) -> OrderService:
    return OrderService(uow_factory=uow_factory)


OrderServiceDep = Annotated[OrderService, Depends(get_order_service)]
```

Router 与 Service 都不承担组装职责，组装只在此处（见规范 §11.1）。

### 5.7 Router（`api/routes/order/order_view.py`）

```python
@router.post("", response_model=OrderResponse, status_code=201)
async def create_order(
    payload: OrderCreateRequest,
    service: OrderServiceDep,
) -> OrderResponse:
    command = CreateOrderCommand(
        product_id=payload.product_id,
        quantity=payload.quantity,
    )
    dto = await service.create_order(command)
    return OrderResponse.model_validate(dto)
```

Router 只做 Schema↔DTO 转换与编排，不写业务逻辑、不 `try/except` 领域异常。

### 5.8 异常映射已由边界层统一处理

`InsufficientStockError`（`ConflictError`）会被全局处理器映射为 409（见规范 §17），Router 无需感知。

### 5.9 测试（见 §6.9 与规范 §18）

用 Fake UoW 单测 Service，断言 `uow.committed`、异常路径未提交、Outbox 已写入。

## 6. 常见配方 Recipes

### 6.1 分页查询

```python
async def list_orders(uow: UnitOfWork, page: int, size: int) -> Page[OrderDTO]:
    total = await uow.orders.count()
    rows = await uow.orders.list(offset=(page - 1) * size, limit=size)
    return Page(
        items=[OrderMapper.to_dto(r) for r in rows],
        total=total,
        page=page,
        size=size,
    )
```

* 统一分页响应结构；`pages = ceil(total / size)`。
* 计数与取数用同一 `where` 条件，避免不一致。

### 6.2 事务边界（只读 vs 写）

```python
# 写用例：显式 commit
async with self._uow_factory() as uow:
    ...
    await uow.commit()

# 只读用例：不写 commit，退出时自动回滚释放
async with self._uow_factory() as uow:
    order = await uow.orders.get_with_items(order_id)
    return OrderMapper.to_dto(order)
```

提前返回也安全：未 `commit` 即回滚（见规范 §12.4）。

### 6.3 ORM → DTO 与关联加载

* 关联数据在 Repository **显式** `selectinload` / `joinedload`，禁止依赖序列化时懒加载（见规范 §13.3）。
* 集合关系优先 `selectinload`，多对一小数据可 `joinedload`，不允许无脑全 `joinedload`。
* 开发/测试对不该懒加载的关系设 `lazy="raise"` 或查询级 `raiseload("*")`，让问题在测试期暴露。

### 6.4 进程内事件（可容忍丢失的副作用）

```python
# 在事务 commit 成功之后发布；失败不阻断主流程（见规范 §14.2 默认策略）
await uow.commit()
await event_bus.publish(OrderCreatedEvent.of(order))
```

仅用于非关键日志、缓存失效、指标等。**关键业务不得只用进程内事件**（见规范 §14.3）。

### 6.5 Transactional Outbox（关键事件）

业务数据与 Outbox 消息在**同一事务**提交（见规范 §14.5）：

```python
async with self._uow_factory() as uow:
    await uow.orders.add(order)
    await uow.outbox.add(OutboxMessage.from_event(OrderCreatedEvent.of(order)))
    await uow.commit()   # 二者原子提交，杜绝双写窗口
```

独立 Worker 领取消息，多实例用 `FOR UPDATE SKIP LOCKED` 防重复领取，按 `aggregate_id` 保序。

### 6.6 消费者幂等（Inbox）

```python
# 去重插入 + 业务逻辑 = 同一事务；靠 (consumer_name, message_id) 唯一约束判重
async with self._uow_factory() as uow:
    try:
        await uow.inbox.mark_processed(consumer, message_id)
    except UniqueViolation:
        return  # 已消费，跳过
    await handle(message)
    await uow.commit()
```

见规范 §14.6。

### 6.7 领域异常 → HTTP

在 Composition Root 注册一次全局处理器（见规范 §17.3）：

```python
_STATUS_MAP = {NotFoundError: 404, ConflictError: 409, ValidationError: 422}
register_domain_exception_handlers(app)   # DomainError → 统一错误响应
```

新增异常只需归类到已有基类，无需改 Router。

### 6.8 依赖生命周期

| 生命周期 | 对象 | 位置 |
| --- | --- | --- |
| 应用级 | Settings、Engine、Redis、HTTP Client、消息连接池 | `lifespan` 初始化/释放 |
| 用例级 | Session、UoW、Repository | `async with uow_factory()` |
| 瞬时 | DTO、Command、值对象、Mapper | 每次调用新建 |

见规范 §11.2。

### 6.9 用 Fake UoW 测 Service

```python
def test_create_order_commits():
    uow = FakeUnitOfWork()
    service = OrderService(uow_factory=lambda: uow)

    dto = await service.create_order(CreateOrderCommand(product_id, quantity=1))

    assert uow.committed is True
    assert uow.outbox.messages          # 事件已随业务写入
```

异常路径断言 `uow.committed is False`（验证回滚语义）。集成测试用"外层事务 + savepoint，测试结束回滚"隔离（见规范 §18.3）。

### 6.10 健康检查与就绪

* `/health/live` 只判进程存活，不碰任何外部依赖（见规范 §6.8）。
* `/health/ready` 只聚合核心依赖；可降级依赖（如 Redis 可回源）不进 `ready`（见规范 §15.4）。
* 后台监控协程周期探测并回写就绪标志，依赖恢复后自动重新接流量。

## 7. 提交前检查清单

开发者自查，评审对照：

**分层与依赖**
- [ ] Service 未创建 Session / Client / Repository 实现，未 import `fastapi`
- [ ] Domain 未依赖框架、ORM、HTTP
- [ ] Router 无业务逻辑、无依赖组装、无事务操作

**事务与数据**
- [ ] 写用例有唯一明确的 `commit` 点；只读用例不提交
- [ ] Repository 未自行 `commit`
- [ ] 关联数据显式加载，未依赖懒加载

**边界隔离**
- [ ] API 响应是 DTO/Schema，未返回 ORM Model
- [ ] Service 抛领域异常而非 `HTTPException`；新异常已归类并可被映射

**可靠性**
- [ ] 关键事件走 Outbox，与业务数据同事务
- [ ] 消费者幂等，有唯一约束兜底
- [ ] 无 `except Exception: pass`，异常有记录与降级

**工程**
- [ ] `ruff format`、`ruff check`、`mypy`、`pytest` 全绿
- [ ] 覆盖提交/回滚与异常路径测试
- [ ] 迁移与代码同 PR，且向后兼容
- [ ] 提交信息符合 Conventional Commits，一次提交一件事
