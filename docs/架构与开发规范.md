# FastAPI 项目架构与开发规范 v1.2（完整版）

> 本文合并 v1.0 基线与后续全部修订/增补（原 v1.1、v1.2），为**单一完整规范**，无需再跨文件查阅。
> 配套《FastAPI 项目开发手册》提供逐层落地的写法示例。

## 版本变更说明

相对 v1.0 基线，本版本引入以下工程约束：

1. 明确依赖注入与对象生命周期；
2. 统一 Unit of Work 事务管理方式；
3. 阻止 ORM 对象跨越应用层边界；
4. 区分同步模块调用、进程内事件和可靠消息；
5. 引入 Transactional Outbox 处理关键事件；
6. 将启动失败、就绪检查和服务降级分级处理；
7. 明确 UoW 采用用例级工厂模式，消除生命周期歧义；
8. 明确 UoW 与模块边界的关系及事务嵌套语义；
9. 补充事件总线默认策略、Outbox 多实例并发与消息追踪；
10. 修正启动分级中 schema 校验与就绪检查的处理方式；
11. 增加优雅停机、领域异常到 HTTP 的边界映射与测试策略。

其中第 1–11 项对应第 11–18 章；对第 4.2、5.1、6.4、6.8 节的修订已就地并入正文。

# 1. 参考项目与取舍

| 参考项目                        | 借鉴内容                                                            | 不直接照搬的部分                   |
| --------------------------- | --------------------------------------------------------------- | -------------------------- |
| FastAPI 官方文档                | `APIRouter` 拆分、依赖注入、路由级依赖、测试依赖覆盖                                | 官方示例偏教学，不包含完整业务分层          |
| Full Stack FastAPI Template | PostgreSQL、Alembic、Pytest、Docker Compose、GitHub Actions、认证和配置管理 | 模型和 CRUD 集中在少数文件，业务增长后容易膨胀 |
| fastapi-best-practices      | 按业务领域组织目录、异步规范、迁移规范、任务队列使用边界、Ruff                               | 部分建议带有作者团队偏好，需要按业务调整       |
| fastapi-best-architecture   | API、Schema、Service、CRUD、Model 三层划分，以及 Celery、监控等企业组件            | 组件较多，不适合项目早期全部引入           |
| Netflix Dispatch            | 按业务模块组织代码的思路                                                    | 项目已于 2025 年归档，只借鉴历史架构思想    |

FastAPI 官方明确支持通过多个 `APIRouter` 组织大型应用，并允许在路由级设置前缀、标签和依赖；测试时可以使用 `app.dependency_overrides` 替换数据库或外部服务依赖。

官方 Full Stack 模板提供 PostgreSQL、SQLModel、Docker Compose、Pytest、Alembic 和 GitHub Actions，但其后端仍以集中式 `models.py`、`crud.py` 和 API 路由为主，更适合作为工程化基础，而不是直接作为复杂领域架构。

社区的 fastapi-best-practices 推荐按领域组织模块，并覆盖异步接口、依赖复用、后台任务、数据库命名、Alembic 和 Ruff 等规范；它明确说明该模块化结构受到 Netflix Dispatch 启发。

# 2. 总体架构决策

推荐采用：

> **模块化单体 + 领域优先组织 + Service/Repository 分层 + 可选 Unit of Work**

先保持一个部署单元，不要在业务边界尚未稳定时拆微服务。

```text
HTTP Request
    ↓
API / Router
    ↓
Application Service
    ↓
Repository / External Client
    ↓
PostgreSQL / Redis / Third-party API
```

核心约束：

```text
API 可以依赖 Service
Service 可以依赖 Repository 接口
Repository 可以依赖 SQLAlchemy
下层不得反向依赖 API
业务模块不得直接访问其他模块的数据库表
```

## 3.1 为什么按业务模块组织

不推荐这种全局分类：

```text
routers/
services/
repositories/
models/
schemas/
```

当系统拥有几十个业务对象后，修改一个订单功能需要同时在五六个目录之间跳转。

推荐：

```text
modules/
├── orders/
├── payments/
└── identity/
```

一个业务功能的大部分代码位于同一个目录，更容易定位、测试和拆分。

# 4. 各层职责

## 4.1 Router

Router 只处理 HTTP 相关内容：

* 路径参数和查询参数；
* Pydantic 请求校验；
* 身份和权限依赖；
* 调用 Service；
* 设置状态码和响应模型。

禁止：

* 直接执行 SQL；
* 在路由中提交事务；
* 编写复杂业务判断；
* 直接调用多个 Repository 编排流程。

```python
@router.post("", response_model=OrderResponse, status_code=201)
async def create_order(
    payload: OrderCreateRequest,
    service: OrderServiceDep,
) -> OrderResponse:
    dto = await service.create_order(payload.to_command())
    return OrderResponse.model_validate(dto)
```

## 4.2 Service

Service 负责：

* 业务规则和用例流程；
* 多个 Repository 的协作；
* 事务边界；
* 领域实体和值对象的创建；
* DTO 转换；
* 领域事件的产生；
* 非关键进程内事件或可靠事件的编排。

Service 禁止：

* 创建数据库 Session；
* 创建 Repository 的具体实现；
* 创建 Redis、HTTP Client、消息生产者等基础设施对象；
* 依赖 FastAPI 的 `Request`、`Response`、`HTTPException`；
* 返回仍然绑定数据库 Session 的 ORM 对象。

Service 可以创建纯业务对象：

```python
order = Order.create(...)
money = Money(amount=amount, currency=currency)
event = OrderCreatedEvent(order_id=order.id)
```

基础设施对象必须由依赖注入层统一组装：

```python
service = OrderService(
    uow_factory=uow_factory,
    event_bus=event_bus,
)
```

带事务的完整用例写法见 §12.2；DTO 转换与 ORM 隔离见 §13。

## 4.3 Repository

Repository 负责数据访问：

* 查询和持久化；
* 屏蔽 SQLAlchemy 查询细节；
* 返回明确的数据对象；
* 不包含业务规则。

Repository 默认不执行 `commit()`：

```python
class SqlAlchemyOrderRepository:
    def __init__(self, session: AsyncSession) -> None:
        self._session = session

    async def add(self, order: Order) -> None:
        self._session.add(order)

    async def get_by_id(self, order_id: UUID) -> Order | None:
        return await self._session.get(Order, order_id)
```

由 Service 或 Unit of Work 决定事务何时提交。

## 4.4 Schema 与 ORM Model

必须区分：

```text
Pydantic Schema：API 输入输出
SQLAlchemy Model：数据库持久化
Application DTO：应用层内部传输（见 §13）
```

禁止直接把 ORM Model 作为公开接口契约。数据库字段变化不应自动改变 API。

推荐命名：

```text
OrderCreateRequest   创建请求
OrderUpdateRequest   更新请求
OrderResponse        单项响应
OrderListResponse    列表响应
OrderFilter          查询条件
OrderDTO             应用层 DTO
```

# 5. 模块边界规范

## 5.1 跨模块调用

模块之间不得直接访问对方的：

* ORM Model；
* Repository；
* 数据库表；
* 内部工具函数；
* 非公开 Service 实现。

模块之间根据一致性需求选择通信方式。

| 场景          | 推荐方式                      |
| ----------- | ------------------------- |
| 查询另一个模块的数据  | 公开 Query Service 或 Facade |
| 需要立即返回结果    | 同步 Application Service    |
| 必须在同一事务中完成  | 同步调用或重新评估模块边界             |
| 非强一致副作用     | 进程内事件                     |
| 必须可靠执行的异步任务 | Outbox + 消息队列             |
| 跨服务长流程      | Saga 或补偿机制                |

允许：

```python
from app.modules.identity.public import IdentityQueryService
```

禁止：

```python
from app.modules.identity.models import User
from app.modules.identity.repository import UserRepository
```

事件不能代替所有同步调用。需要立即获得结果、校验业务条件或保证强一致性的操作，应使用明确的同步接口。

## 5.2 Shared 目录

`shared` 只能存放不包含具体业务语义的代码，例如：

* 分页结构；
* 通用时间处理；
* 基础类型；
* ID 生成器；
* 通用协议。

禁止把无法分类的代码全部放入 `shared/utils.py`。

## 5.3 是否需要纯领域模型

项目初期可以让 SQLAlchemy Model 同时承载部分领域行为：

```python
order.cancel()
order.confirm()
```

出现以下情况后，再将领域对象与 ORM Model 分离：

* 同一业务对象存在复杂状态机；
* 业务规则需要脱离数据库测试；
* 一个聚合包含多个实体和值对象；
* 持久化结构与业务模型差异明显。

不要为了形式上的 Clean Architecture 提前创建大量接口和映射器。

# 6. 开发规范

## 6.1 Python 与依赖管理

统一使用：

```text
pyproject.toml
uv
Ruff
Pyright 或 Mypy
Pytest
pre-commit
```

规范：

1. 锁定 Python 次版本。
2. 提交 `uv.lock`。
3. 生产依赖和开发依赖分组管理。
4. 禁止手工维护多份内容重复的 requirements 文件。
5. CI 必须从锁文件安装依赖。

## 6.2 代码风格

所有公开函数必须标注参数和返回类型：

```python
async def get_order(order_id: UUID) -> Order | None:
    ...
```

禁止：

```python
async def get_order(id):
    ...
```

其他要求：

* 使用绝对导入；
* 禁止 `from module import *`；
* 一个函数原则上只承担一个职责；
* 业务函数优先控制在约 40 行以内；
* 文件名、变量名使用 `snake_case`；
* 类名使用 `PascalCase`；
* 常量使用 `UPPER_SNAKE_CASE`；
* 不使用含糊名称，如 `data`、`obj`、`handle()`；
* 注释解释“为什么”，不要重复代码含义。

## 6.3 API 规范

统一版本前缀：

```text
/api/v1
```

路径使用资源名词：

```text
GET    /api/v1/orders
POST   /api/v1/orders
GET    /api/v1/orders/{order_id}
PATCH  /api/v1/orders/{order_id}
DELETE /api/v1/orders/{order_id}
```

动作型接口只用于无法自然表达为资源的操作：

```text
POST /api/v1/orders/{order_id}/cancel
POST /api/v1/orders/{order_id}/confirm
```

响应规则：

* 创建成功：`201`
* 删除成功且无响应体：`204`
* 参数错误：`422`
* 未认证：`401`
* 无权限：`403`
* 资源不存在：`404`
* 状态冲突：`409`

统一错误结构：

```json
{
  "error": {
    "code": "ORDER_INSUFFICIENT_STOCK",
    "message": "Insufficient product stock",
    "details": {
      "product_id": "..."
    },
    "request_id": "..."
  }
}
```

错误码必须稳定，前端不能依赖自然语言错误信息。领域异常到 HTTP 响应的转换机制见 §17。

## 6.4 数据库规范

**命名**统一使用小写下划线：

```text
user
order
order_item
payment_transaction
```

**索引与约束：**

1. 所有外键建立索引。
2. 经常用于过滤、排序的字段按查询计划建立索引。
3. 唯一性约束必须由数据库保证。

**数据类型：**

4. 时间统一存储为 UTC。
5. 金额使用 `Decimal` 或数据库 `NUMERIC`，禁止使用浮点数。

**Session 与事务：**

6. 一个数据库 Session 只能服务于一个请求或一个独立任务，禁止全局共享。
7. 同一个 `AsyncSession` 不得在多个并发协程之间共享。
8. Repository 默认不得执行 `commit()`。
9. Service 决定业务事务边界，一个业务用例尽量对应一个事务。
10. Unit of Work 负责 Session 的提交、回滚和释放。
11. 项目必须在“显式提交”和“成功退出自动提交”之间选择一种策略，默认推荐显式提交。

**ORM/DTO 与懒加载：**

12. ORM 到 DTO 的转换必须在 Session 有效期间完成。
13. 异步 ORM 查询必须避免不可控的隐式懒加载。
14. 大批量处理禁止逐行查询，避免 N+1；测试环境应主动暴露 N+1 和懒加载问题。

**分页：**

15. API 分页必须设置最大页大小。

**关系加载策略：**

* 多对一、小数据关系可考虑 `joinedload()`；
* 一对多集合通常优先考虑 `selectinload()`；
* 需要阻止意外查询时可使用 `raiseload()`；
* 禁止无依据地对所有关系统一使用 `joinedload()`。

Session、UoW 与 DTO 隔离的完整规范见 §11–§13。

## 6.5 Alembic 迁移规范

每次数据库结构变更必须提交迁移文件。

迁移文件命名：

```text
2026-07-02_add_order_status_index.py
```

要求：

* `upgrade()` 和 `downgrade()` 均可执行；
* 迁移内容必须静态、可审核；
* 禁止在应用启动时自动生成迁移；
* 禁止修改已经在生产执行过的迁移文件；
* 大表字段修改必须评估锁表时间；
* 数据迁移和结构迁移复杂时分开执行。

官方模板同样将 Alembic 迁移纳入版本控制，并要求模型变化后生成和执行迁移。

## 6.6 异步规范

使用 `async def` 的场景：

* 异步数据库驱动；
* HTTP 客户端；
* Redis；
* 消息队列；
* 文件或对象存储的异步 SDK。

禁止在事件循环中直接执行：

* `requests`；
* 同步数据库查询；
* 大文件压缩；
* 图片处理；
* 大规模数据计算；
* 长时间同步 SDK 调用。

短暂同步调用放入线程池。耗时、需要重试或不能丢失的任务进入 Celery、Dramatiq、Arq 等任务队列。

`BackgroundTasks` 只用于短小且允许丢失的非关键任务。社区最佳实践也指出，它运行在当前 Worker 进程中，没有持久化、重试和任务可见性，不应承担关键任务。

## 6.7 测试规范

测试分为四层：

```text
unit         Service 和纯业务逻辑
integration  Repository、数据库、Redis、外部客户端适配器
api          FastAPI 路由和依赖
e2e          完整关键业务流程
```

目录示例：

```text
tests/
├── unit/modules/orders/test_order_service.py
├── integration/modules/orders/test_order_repository.py
├── api/test_orders_api.py
└── e2e/test_order_payment_flow.py
```

规则：

* 单元测试不得连接真实数据库；
* 集成测试使用独立测试数据库；
* 每个测试自行创建数据；
* 禁止依赖测试执行顺序；
* 外部 API 必须 Mock 或使用本地 Stub；
* 时间、UUID 和随机数应可替换；
* 修复缺陷时先补充回归测试；
* CI 中禁止使用开发人员本地数据库。

FastAPI 官方提供 `dependency_overrides`，可在测试中将数据库、认证或外部服务依赖替换成测试实现。分层测试的详细做法（Fake UoW、事务回滚隔离、事件与幂等测试）见 §18。

## 6.8 日志与可观测性

日志使用结构化 JSON，至少包含：

```text
timestamp
level
service
environment
request_id
trace_id
user_id
method
path
status_code
duration_ms
```

要求：

* 每个请求生成或透传 `request_id`；
* 禁止记录密码、Token、Cookie 和银行卡信息；
* 异常日志保留堆栈；
* 业务异常记录为 `warning`，未知异常记录为 `error`；
* 指标至少包括请求量、延迟、错误率和数据库连接池状态。

健康检查必须分为三个端点：

```text
/health/live
/health/ready
/health/startup
```

### `/health/live`

只判断进程是否仍在运行。

不应在该接口中访问：

* 数据库；
* Redis；
* 消息队列；
* 第三方服务。

外部依赖异常不应导致存活检查失败，否则可能引起实例反复重启。

### `/health/ready`

判断实例当前是否具备接收业务流量的条件。

可检查：

* 核心数据库；
* 必要缓存；
* 必要消息基础设施；
* 数据库 Schema 版本；

# 11. 依赖注入与生命周期管理规范

## 11.1 Composition Root

项目必须设置明确的依赖组装入口，即 Composition Root。

推荐位置：

```text
app/
├── bootstrap.py
├── api/dependencies.py
└── core/lifespan.py
```

依赖组装代码负责：

* 创建 Repository；
* 创建 Unit of Work；
* 创建 Service；
* 注入配置；
* 注入外部 Client；
* 注入消息生产者；
* 控制对象生命周期。

Router 和 Service 不得承担依赖组装职责。

## 11.2 生命周期分类

依赖至少划分为三种生命周期。

### 应用级

整个应用进程共享：

* Settings；
* HTTP Client；
* Redis Client；
* 数据库 Engine；
* 消息队列连接池；
* 指标和追踪对象。

应用级依赖通常在 `lifespan` 中初始化和释放。

### 请求级

每次请求独立创建：

* 包含请求上下文的 Service；
* 当前用户上下文；
* Request ID、追踪上下文等请求元数据。

FastAPI 默认会在一次请求内缓存同一个依赖函数的返回值。请求级依赖（如当前用户上下文）在一次请求内复用同一实例。

### 用例级（Session 与 UoW 的默认作用域）

Session 与 UoW 的作用域必须二选一，不能混用，否则会出现"注入的实例被反复进入/关闭"的歧义：

* **默认：用例级工厂模式（推荐）**
  DI 注入的是应用级的 `session_factory`（`async_sessionmaker`），Service 每次通过 `async with uow_factory() as uow` 开启一个用例级事务作用域。一次业务操作 = 一个 Session = 一个事务，进入即建、退出即关。这是本规范的默认模型，下文 12 章的所有示例都基于它。

* **可选：请求级共享 Session**
  仅当一次请求内多个用例必须落在**同一个事务**里时，才把 Session 提升为请求级共享（`Depends` 缓存同一 Session Provider）。此时 Service 不得再自行 `async with` 该 Session，事务边界由请求层统一控制。

不要把"请求级 Session"和"用例级 UoW 事务"当成同一件事：前者是"整请求一个事务"，后者是"每用例一个事务"，二者语义不同，必须显式选择一种。

### 瞬时对象

每次调用都可以重新创建：

* DTO；
* Command；
* 值对象；
* Mapper；
* 无状态领域服务。

## 11.3 配置和值提供者

静态配置优先通过类型化 Settings 注入：

```python
from typing import Annotated

from fastapi import Depends

from app.core.config import Settings, get_settings


SettingsDep = Annotated[Settings, Depends(get_settings)]
```

依赖工厂：

```python
def get_order_service(
    unit_of_work: UnitOfWorkDep,
    settings: SettingsDep,
) -> OrderService:
    return OrderService(
        unit_of_work=unit_of_work,
        run_mode=settings.run_mode,
    )
```

只有真正依赖当前应用实例或请求上下文时，才使用 `Request`：

```python
def get_request_id(request: Request) -> str:
    return request.state.request_id
```

禁止在 Service 中读取：

```python
request.app.state
os.environ
fastapi.Request
```

## 11.4 工厂函数的同步与异步选择

没有异步操作的依赖工厂使用普通 `def`：

```python
def get_order_repository(
    session: DbSessionDep,
) -> OrderRepository:
    return SqlAlchemyOrderRepository(session)
```

只有需要执行异步初始化时才使用 `async def`：

```python
async def get_external_context(
    client: ExternalClientDep,
) -> ExternalContext:
    return await client.load_context()
```

不要为了保持形式统一，将所有依赖工厂都声明为异步函数。

## 11.5 禁止事项

禁止在 Router 中执行：

```python
repo = SqlAlchemyOrderRepository(session)
service = OrderService(repo)
```

禁止在 Service 中执行：

```python
session = AsyncSession(...)
redis = Redis(...)
client = httpx.AsyncClient(...)
```

允许在 Service 中执行：

```python
order = Order.create(...)
event = OrderCreatedEvent(...)
dto = OrderMapper.to_dto(order)
```

# 12. Unit of Work 与事务边界规范

## 12.1 事务责任划分

事务职责必须明确：

```text
DI 层
负责创建和释放 UoW

Service 层
决定事务范围和提交时机

UoW
执行 commit、rollback、close

Repository
执行数据读写，不提交事务
```

Router 不得调用：

```python
session.commit()
session.rollback()
uow.commit()
```

## 12.2 默认采用显式提交

推荐的 Service 写法（用例级工厂模式）：

```python
from collections.abc import Callable


class OrderService:
    def __init__(
        self,
        uow_factory: Callable[[], UnitOfWork],
    ) -> None:
        self._uow_factory = uow_factory

    async def create_order(
        self,
        command: OrderCreate,
    ) -> OrderDTO:
        async with self._uow_factory() as uow:
            product = await uow.products.get_required(
                command.product_id
            )

            if product.stock < command.quantity:
                raise InsufficientStockError(product.id)

            product.reduce_stock(command.quantity)

            order = Order.create(
                product_id=product.id,
                quantity=command.quantity,
            )

            await uow.orders.add(order)

            dto = OrderMapper.to_dto(order)
            await uow.commit()

            return dto
```

Service 注入的是 **UoW 工厂**而不是 UoW 实例。每次 `async with self._uow_factory()` 都会新建一个 Session 和事务作用域，进入即建、退出即关。这样做的原因：

* 避免注入单个 request-scoped 实例后，被多个用例反复 `async with` 而重复进入/关闭同一对象；
* 每个用例的事务边界彼此独立，互不污染；
* 与 11.2 的"用例级作用域"保持一致。

如果项目改用"请求级共享 Session"模型（见 11.2 可选项），则 Service 不再持有工厂、也不再写 `async with`，事务边界由请求层统一控制。两种模型只能选其一。

显式提交的优点：

* 提交点清晰；
* 查询方法不会意外提交；
* 提前返回时行为明确；
* 代码审查时容易识别事务范围。

## 12.3 UoW 抽象接口

```python
from types import TracebackType
from typing import Protocol, Self


class UnitOfWork(Protocol):
    orders: OrderRepository
    products: ProductRepository
    outbox: OutboxRepository

    async def __aenter__(self) -> Self:
        ...

    async def __aexit__(
        self,
        exc_type: type[BaseException] | None,
        exc_value: BaseException | None,
        traceback: TracebackType | None,
    ) -> None:
        ...

    async def commit(self) -> None:
        ...

    async def rollback(self) -> None:
        ...
```

`orders`、`products`、`outbox` 等仓储属性仅在进入上下文（`__aenter__`）之后才可用，它们绑定于本次上下文创建的 Session。在进入上下文之前访问这些属性属于错误用法，实现应保证此时抛出明确异常而不是返回未初始化对象。因此 UoW 只能通过 `async with` 使用，不应把它的仓储在上下文之外传递或缓存。

## 12.4 SQLAlchemy UoW 行为

当项目选择显式提交策略时，`__aexit__()` 必须遵循：

* 发生异常时回滚；
* 未显式提交时回滚；
* 无论成功或失败都关闭 Session；
* 不得在退出时隐式提交。

```python
class SqlAlchemyUnitOfWork:
    def __init__(
        self,
        session_factory: async_sessionmaker[AsyncSession],
    ) -> None:
        self._session_factory = session_factory
        self._session: AsyncSession | None = None
        self._committed = False

    async def __aenter__(self) -> "SqlAlchemyUnitOfWork":
        self._session = self._session_factory()

        self.orders = SqlAlchemyOrderRepository(self._session)
        self.products = SqlAlchemyProductRepository(self._session)
        self.outbox = SqlAlchemyOutboxRepository(self._session)

        return self

    async def commit(self) -> None:
        self._require_session()
        await self._session.commit()
        self._committed = True

    async def rollback(self) -> None:
        if self._session is not None:
            await self._session.rollback()

    async def __aexit__(
        self,
        exc_type: type[BaseException] | None,
        exc_value: BaseException | None,
        traceback: TracebackType | None,
    ) -> None:
        if self._session is None:
            return

        try:
            if exc_type is not None or not self._committed:
                await self._session.rollback()
        finally:
            await self._session.close()
            self._session = None

    def _require_session(self) -> None:
        if self._session is None:
            raise RuntimeError("Unit of Work is not active")
```

## 12.5 UoW 的使用边界

以下场景适合使用 UoW：

* 一个用例修改多个聚合；
* 一个业务操作写入多个 Repository；
* 数据变更和 Outbox 消息必须原子提交；
* 需要明确测试事务提交或回滚行为。

简单只读项目可以暂时使用请求级 Session，不必为了形式增加 UoW。

## 12.6 UoW 与模块边界

UoW 是**事务一致性边界**，不是"全局仓储容器"。它必须限定在单一限界上下文（模块）内部，只聚合本模块的仓储。

* 一个模块内部：UoW 可同时暴露该模块的多个聚合仓储（如订单模块的 `orders`、`order_items`），在一个事务内原子提交。
* 跨模块：不应存在一个横跨多个模块仓储的"上帝 UoW"。12.3 中 `orders`、`products`、`outbox` 属于同一个"下单"限界上下文才成立；若 `products` 实际归属独立的商品模块，则该结构违反 5.1 的模块隔离。

当出现"两个模块的写操作必须在同一事务内提交"时，这是**边界划分错误的信号**，而不是引入跨模块 UoW 的理由。应按以下顺序处理：

1. 重新评估边界：这两块数据是否本就属于同一个聚合/模块，应当合并；
2. 若确属不同模块，改用最终一致性：本模块事务内写业务数据 + Outbox 消息，另一模块异步消费（见 14 章）；
3. 只有在同一数据库、同一 Schema 且边界确实无法再调整时，才允许由更上层的应用服务显式协调一个共享事务，并在文档中记录这一妥协及其原因。

Outbox 仓储是例外：它可以出现在任意模块的 UoW 中，因为它服务于"业务写入与消息写入原子提交"，本质上仍属于当前模块的事务。

## 12.7 事务嵌套与用例互调

当一个用例（已在 `async with uow` 中）需要调用另一个用例时，禁止让被调用方再开一个独立 UoW / 事务，否则会出现两个互不感知的事务，破坏原子性。约定：

* **拆分领域逻辑与事务边界**：把可复用的业务逻辑写成不自带事务的领域方法/协作方法，由最外层用例统一开启 UoW 并传入，内层只使用传入的 `uow`，不自行 `commit`。
* **一个请求内只应有一个"顶层"事务边界**，由发起用例负责 `commit` / `rollback`。
* 确需局部回滚而不影响整体事务时，使用 Session 的 **savepoint（嵌套事务）**：

```python
async with uow.session.begin_nested():
    await risky_step(uow)
```

* 禁止在被调用的内层方法里调用 `uow.commit()`；提交权只属于顶层用例。

# 13. ORM 上下文隔离与 DTO 规范

## 13.1 ORM 泄漏红线

API 层不得接收仍然依赖 Session 的 ORM 对象。

禁止：

```python
async def get_order(...) -> Order:
    return await repository.get_by_id(order_id)
```

禁止直接将 ORM Model 作为接口响应：

```python
@router.get("/{order_id}")
async def get_order(...) -> Order:
    return await service.get_order(order_id)
```

主要风险：

* 序列化阶段触发懒加载；
* Session 已关闭后访问关联字段；
* 出现 `MissingGreenlet`；
* API 契约与数据库结构耦合；
* 无意暴露内部字段；
* 难以控制查询数量。

## 13.2 DTO 边界

推荐数据流：

```text
SQLAlchemy ORM Model
        ↓
Application Mapper
        ↓
Application DTO
        ↓
Router
        ↓
API Response Schema
```

Application DTO 不依赖 FastAPI：

```python
from dataclasses import dataclass
from decimal import Decimal
from uuid import UUID


@dataclass(frozen=True, slots=True)
class OrderDTO:
    id: UUID
    status: str
    total_amount: Decimal
    items: tuple["OrderItemDTO", ...]
```

Service 在 Session 有效期间完成转换：

```python
async def get_order_detail(
    self,
    order_id: UUID,
) -> OrderDTO:
    async with self._uow_factory() as uow:
        order = await uow.orders.get_with_items(order_id)
        return OrderMapper.to_dto(order)
```

Router 转换为 API Schema：

```python
@router.get("/{order_id}", response_model=OrderResponse)
async def get_order(
    order_id: UUID,
    service: OrderServiceDep,
) -> OrderResponse:
    dto = await service.get_order_detail(order_id)
    return OrderResponse.model_validate(dto)
```

小型项目允许 Service 直接返回 Pydantic DTO，但该 DTO 必须属于 Application 层，不能直接依赖 HTTP 状态码、Request 或 Response。

## 13.3 关联数据加载

Repository 必须显式声明查询所需的关联数据：

```python
statement = (
    select(Order)
    .options(selectinload(Order.items))
    .where(Order.id == order_id)
)
```

不得依赖序列化过程自动加载关联字段。

开发和测试阶段可对不允许懒加载的关系使用：

```python
lazy="raise"
```

或查询级：

```python
raiseload("*")
```

## 13.4 提交后的属性访问

异步 Session 推荐设置：

```python
async_sessionmaker(
    bind=engine,
    expire_on_commit=False,
)
```

但 `expire_on_commit=False` 不能代替 DTO 转换和显式加载。

它只能降低提交后访问基础属性时发生隐式查询的概率，不能保证所有关系都已经加载。

# 14. 模块解耦、事件总线与 Outbox

## 14.1 事件使用原则

事件适合表达已经发生的业务事实：

```text
OrderCreated
PaymentSucceeded
UserRegistered
InventoryReserved
```

事件名称使用过去式。

不推荐：

```text
CreateOrder
SendEmail
UpdatePoints
```

以上名称更接近 Command，而不是 Event。

## 14.2 进程内事件总线

进程内 Event Bus 适用于：

* 非关键日志；
* 本地缓存失效；
* 指标记录；
* 可容忍丢失的扩展逻辑；
* 单体应用内的轻量解耦。

接口示例：

```python
from typing import Protocol, TypeVar


EventT = TypeVar("EventT")


class EventBus(Protocol):
    async def publish(self, event: object) -> None:
        ...
```

Handler 注册应集中管理：

```python
event_bus.register(
    OrderCreatedEvent,
    update_order_metrics,
)
```

必须明确：

* Handler 串行还是并行；
* 一个 Handler 失败是否阻止其他 Handler；
* 是否向调用方传播异常；
* 是否允许重复处理；
* 是否记录处理结果。

本规范给出如下**推荐默认值**（项目可按需覆盖，但必须显式声明）：

* **发布时机**：在事务 `commit()` 成功之后发布进程内事件；进程内事件本就用于可容忍丢失的副作用，不参与业务事务。
* **执行方式**：默认串行执行 Handler，保证可预测的顺序和更简单的调试；仅在 Handler 明确无副作用依赖时才并行。
* **失败隔离**：单个 Handler 失败不阻断其他 Handler，也不向调用方传播异常，仅记录日志与告警。因为进程内事件承载的是非关键逻辑；任何"失败必须影响主流程"的逻辑都不应走进程内事件，而应改为同步调用或 Outbox。
* **重复处理**：进程内事件不保证投递次数，Handler 应尽量幂等。
* **结果记录**：每个 Handler 的成功/失败都应记录，便于排查"事件已发布但副作用未生效"。

## 14.3 关键业务禁止只使用进程内事件

以下业务不得只依赖进程内 Event Bus：

* 支付结果处理；
* 积分入账；
* 库存最终扣减；
* 财务记账；
* 退款；
* 必须发送的通知；
* 跨实例任务；
* 需要重试的任务。

进程退出、Worker 重启或实例崩溃都可能导致事件丢失。

## 14.4 禁止提交后直接发布关键事件

不可靠写法：

```python
await uow.commit()
await event_bus.publish(event)
```

该代码存在双写窗口：

```text
数据库提交成功
进程在消息发布前崩溃
事件永久丢失
```

关键事件必须使用 Transactional Outbox。

## 14.5 Transactional Outbox

业务数据和 Outbox 消息必须在同一个数据库事务中提交：

```python
async with self._uow_factory() as uow:
    order = Order.create(...)

    await uow.orders.add(order)
    await uow.outbox.add(
        OutboxMessage.from_event(
            OrderCreatedEvent(
                order_id=order.id,
                user_id=order.user_id,
            )
        )
    )

    dto = OrderMapper.to_dto(order)
    await uow.commit()

return dto
```

Outbox 表建议包含：

```text
id
event_type
aggregate_type
aggregate_id
payload
status
created_at
published_at
retry_count
next_retry_at
last_error
```

独立 Worker 负责：

1. 查询待发送消息；
2. 发布到消息队列；
3. 标记为已发布；
4. 失败时记录原因并重试；
5. 超过阈值后进入死信处理。

### 多实例并发

服务多实例部署时，多个 Worker 会并发轮询同一张 Outbox 表，必须防止同一条消息被重复领取：

* 领取消息使用行级锁跳过已锁定行：

```sql
SELECT * FROM outbox
WHERE status = 'pending' AND next_retry_at <= now()
ORDER BY created_at
LIMIT :batch_size
FOR UPDATE SKIP LOCKED;
```

* 或引入 `locked_by` / `locked_until` 列做租约（lease），Worker 崩溃后租约到期消息可被重新领取。
* 至少一次投递是常态，重复由消费者幂等（14.6）兜底。

### 消息顺序

轮询模型默认不保证跨消息的全局顺序。当同一聚合的事件存在先后依赖（如 `OrderCreated` 必须先于 `OrderPaid`）时：

* 按 `aggregate_id` 分区/分组发送，保证同一聚合内有序；
* 消费者侧对乱序具备容忍能力（丢弃过期版本或缓存等待）；
* 不要假设不同聚合之间有序。

### 保留与清理

已发布消息会持续累积，必须有清理策略：

* 已发布消息保留一段时间（用于排查与审计）后归档或删除；
* 通过定时任务按 `published_at` 清理；
* 死信消息单独保留并告警，不随普通清理删除。

### Worker 部署形态

* Outbox Worker 应作为**独立进程/独立部署单元**运行，与 API 进程解耦，避免 API 扩缩容影响消息发送，也避免 Web 进程重启中断发送。
* 它复用应用级的 `engine` / `session_factory`，但拥有独立的生命周期与健康检查。
* 小型场景可临时以 API 进程内后台任务运行，但必须在文档中标注为过渡方案，并注意多实例下的并发领取问题。

## 14.6 消费者幂等性

消息系统通常只能可靠实现“至少一次”投递，因此消费者必须支持重复消息。

推荐建立 Inbox 或消费记录表：

```text
consumer_name
message_id
processed_at
```

处理流程：

```text
在同一事务内：
  插入消费记录 (consumer_name, message_id)
      ↓ 命中唯一约束冲突 → 说明已消费，回滚并跳过
  执行业务逻辑
      ↓
  提交事务
```

关键约束（否则并发下幂等失效）：

* `(consumer_name, message_id)` 必须建**唯一约束**；
* **消费记录的写入必须与业务逻辑在同一个事务内提交**。若先查后写分成两步、或与业务逻辑分属不同事务，两条并发投递会同时通过"未消费"检查，导致重复处理。正确做法是直接尝试插入消费记录，靠唯一约束冲突来判定"已消费"。

业务唯一约束（如订单号唯一）也应作为最终防线。

## 14.7 事件契约管理

事件结构一旦被其他模块或服务消费，就属于公开契约。

要求：

* 事件包含唯一 `event_id`；
* 包含 `occurred_at`；
* 包含 Schema 版本；
* 包含 `correlation_id`（贯穿整条业务链路，用于跨服务追踪）；
* 包含 `causation_id`（指向直接触发本事件的上一个消息/事件 ID，用于还原因果链）；
* 新增字段优先保持向后兼容；
* 消费者遇到更高的 `schema_version` 时应能安全降级处理或明确拒绝，不得直接崩溃；
* 禁止直接传输 ORM 对象；
* Payload 使用可序列化的基础类型；
* 事件消费者不得依赖生产模块的内部 Model。

示例：

```python
class OrderCreatedEvent(BaseModel):
    event_id: UUID
    schema_version: int = 1
    occurred_at: datetime
    correlation_id: UUID
    causation_id: UUID | None = None
    order_id: UUID
    user_id: UUID
    total_amount: Decimal
```

# 15. 启动阶段 Fail-Fast 与降级策略

## 15.1 分级原则

不是所有依赖异常都应阻止进程启动。

故障必须分为三类：

### A 类：程序不变量错误

必须阻止启动：

* 必填配置缺失；
* Secret 格式错误；
* DSN 格式无效；
* 不支持的运行模式；
* 数据库 Schema 版本不兼容；
* 必须加载的本地资源不存在；
* 核心依赖对象无法构造；
* 安全配置明显无效。

### B 类：核心基础设施暂时不可用

通常不必反复终止进程，但 readiness 必须失败：

* 数据库暂时不可访问；
* Redis 暂时不可访问且业务强依赖 Redis；
* 必需消息基础设施暂时不可访问；
* 关键远程配置服务暂时不可用。

应用进程可以保持运行并持续恢复连接，但在恢复前不得接收业务流量。

### C 类：非核心依赖不可用

应用可以降级运行：

* 邮件服务；
* 短信服务；
* 非必要推荐服务；
* 非关键分析平台；
* 可选第三方接口。

系统应：

* 记录错误；
* 触发告警；
* 返回明确的降级结果；
* 使用重试、Outbox 或任务队列恢复；
* 不得让整个应用启动失败。

## 15.2 Lifespan 职责

`lifespan` 负责：

* 校验关键配置；
* 初始化连接池和共享 Client；
* 注册依赖；
* 启动必要后台组件；
* 释放资源。

示例：

```python
from collections.abc import AsyncIterator
from contextlib import asynccontextmanager

from fastapi import FastAPI


@asynccontextmanager
async def app_lifespan(
    app: FastAPI,
) -> AsyncIterator[None]:
    settings = load_settings()
    validate_critical_settings(settings)

    database = create_database(settings.database_url)
    redis = create_redis_client(settings.redis_url)
    http_client = create_http_client()

    app.state.settings = settings
    app.state.database = database
    app.state.redis = redis
    app.state.http_client = http_client
    app.state.readiness = ReadinessState()

    await check_schema_or_fail(database, app.state.readiness)

    try:
        yield
    finally:
        await http_client.aclose()
        await redis.aclose()
        await database.dispose()
```

关键区分：启动阶段访问数据库时，必须把"**连不上**"和"**连上了但 Schema 版本不兼容**"分开处理，否则 DB 临时抖动会被误判为不可恢复错误而阻止启动，违反 15.1 的 B 类原则。

```python
async def check_schema_or_fail(
    database: Database,
    readiness: "ReadinessState",
) -> None:
    try:
        await asyncio.wait_for(
            database.check_connection(),
            timeout=3,
        )
    except (ConnectionError, asyncio.TimeoutError) as exc:
        # B 类：DB 暂时不可用 —— 不阻止启动，
        # readiness 保持未就绪，由后台监控持续重连（见 15.4）。
        logger.warning("database not reachable at startup: %s", exc)
        readiness.database_ready = False
        return

    version = await database.get_schema_version()
    if not is_compatible(version):
        # A 类：Schema 版本不兼容 —— 程序不变量错误，必须阻止启动。
        raise IncompatibleSchemaError(version)

    readiness.database_ready = True
```

即：能连上但版本不对 → A 类，直接 `raise` 阻止启动；连不上 → B 类，记录并返回，交给就绪检查与后台重连处理。

## 15.3 外部依赖检查限制

启动检查必须设置：

* 超时时间；
* 最大重试次数；
* 指数退避；
* 明确日志；
* 不记录 Secret；
* 不执行昂贵查询。

禁止：

```python
await third_party_api.perform_real_business_request()
```

推荐：

```python
await asyncio.wait_for(
    database.check_connection(),
    timeout=3,
)
```

## 15.4 Readiness 状态

应用应维护可观察的就绪状态。`ready` 只应聚合**核心（B 类）依赖**；可降级（C 类）依赖不得参与 `ready` 判定，否则会与 15.5 的降级策略冲突（例如 Redis 可回源数据库，就不该因 Redis 不可用而拒绝全部流量）：

```python
@dataclass
class ReadinessState:
    # 核心依赖：影响 ready
    database_ready: bool = False
    messaging_ready: bool = False
    # 可降级依赖：仅用于观测与告警，不影响 ready
    redis_ready: bool = False

    @property
    def ready(self) -> bool:
        # 仅聚合核心依赖
        return self.database_ready and self.messaging_ready
```

哪些依赖属于"核心"由 15.5 的降级表决定：失败后"拒绝业务流量"的进入 `ready`，失败后"降级/回源/重试"的不进入。

### 就绪状态的恢复

15.1 的 B 类要求"进程保持运行并持续恢复连接"，这意味着必须有一个**后台监控协程**周期性探测核心依赖并回写这些标志位，而不是只在启动时设置一次：

```python
async def readiness_monitor(
    state: ReadinessState,
    database: Database,
    interval: float = 5.0,
) -> None:
    while True:
        try:
            await asyncio.wait_for(
                database.check_connection(),
                timeout=3,
            )
            state.database_ready = True
        except Exception:
            state.database_ready = False
        await asyncio.sleep(interval)
```

* 该监控在 `lifespan` 内作为后台任务启动，并在关闭时取消；
* 依赖恢复后 `ready` 自动变回 True，实例重新接收流量，无需重启进程；
* 探测必须带超时、轻量，禁止执行昂贵业务查询（见 15.3）。

## 15.5 降级策略

每个外部依赖必须在设计阶段定义失败行为：

| 依赖       | 失败策略                |
| -------- | ------------------- |
| 主数据库     | readiness 失败，拒绝业务流量 |
| Redis 缓存 | 回源数据库或关闭缓存          |
| 消息队列     | 写入 Outbox，等待恢复      |
| 邮件服务     | 任务重试，不阻塞主流程         |
| 支付服务     | 返回明确失败，不假定支付成功      |
| 推荐服务     | 返回默认结果              |
| 日志平台     | 本地标准输出兜底            |

禁止捕获异常后静默忽略：

```python
try:
    await client.send(...)
except Exception:
    pass
```

必须记录异常并执行明确的恢复或降级策略。

## 15.6 优雅停机

进程收到 `SIGTERM`（如滚动发布、缩容、K8s Pod 终止）后不应立即硬退出，否则会中断在途请求、丢失正在处理的消息。停机必须有序：

```text
收到 SIGTERM
    ↓
就绪探针置为未就绪（停止接收新流量）
    ↓
等待负载均衡摘除本实例（宽限期）
    ↓
排空在途请求 / 等待正在处理的消息完成
    ↓
停止后台任务（就绪监控、Outbox Worker 等）
    ↓
关闭连接：消息生产者 → Redis → 数据库 Engine
    ↓
进程退出
```

要点：

* `lifespan` 的 `finally` 段负责释放资源，其关闭顺序应与初始化相反，且在请求排空之后执行；
* 主动把 readiness 置为未就绪并保留**宽限期**（如几秒），给负载均衡/服务发现摘除节点的时间，避免摘除前新请求打进来；
* 后台任务（就绪监控、Outbox Worker）要能响应取消信号并等待其干净退出；
* 为整个停机流程设置**总超时**，超时后强制退出，避免卡死在无法关闭的连接上；
* 部署侧应配置合理的 `terminationGracePeriod`，与应用宽限期匹配。

# 16. 架构复杂度控制

新增 UoW、DTO、Event Bus 和 Outbox 后，团队必须防止过度抽象。

## 16.1 可以暂缓的设计

以下项目可以暂时不引入 UoW：

* 只读 API；
* 单表 CRUD；
* 没有跨 Repository 事务；
* 没有可靠事件需求。

以下项目可以暂时不引入 Event Bus：

* 只有一个业务模块；
* 模块间调用关系简单；
* 所有操作都要求同步返回；
* 团队尚未建立事件监控能力。

以下项目可以暂时不引入 Outbox：

* 事件允许丢失；
* 事件仅用于本地指标；
* 事件不改变关键业务状态。

## 16.2 必须引入的信号

出现以下情况时，应引入 UoW：

* 一个用例修改多张表；
* 多个 Repository 必须原子提交；
* 事务提交和回滚逻辑开始重复；
* 需要同时保存业务数据和 Outbox。

出现以下情况时，应引入 Outbox：

* 数据库成功但消息丢失会造成状态不一致；
* 消息必须重试；
* 服务运行多个实例；
* 事件驱动积分、库存、支付或通知；
* 需要追踪消息发送状态。

## 16.3 目标态与演进路径

需要澄清：本章 16.1 / 16.2 与下面的目标态并不矛盾。**默认起点是最小集**，其余能力按 16.2 的信号出现时**逐步引入**，而不是一开始全部铺开。

**默认最小起点**（任何项目一开始就应具备）：

```text
FastAPI 原生 Depends
+ 明确 Composition Root
+ 用例级工厂 Session（或简单场景下的请求级 Session）
+ ORM 到 DTO 隔离
+ 领域异常到 HTTP 的边界映射（见第 17 章）
+ 分级 Fail-Fast + Liveness / Readiness / Startup 分离
```

**按信号演进引入**（满足 16.2 对应信号时才加）：

```text
显式提交 UoW        ← 出现跨仓储原子事务
同步公开模块接口     ← 出现跨模块调用
非关键进程内事件     ← 出现可容忍丢失的副作用解耦
关键事件 Transactional Outbox  ← 出现"DB 成功但消息丢失会导致状态不一致"
消费者幂等 / Inbox   ← 引入消息队列后
```

**完整目标态**（成熟的中型业务系统最终形态）即上述两部分之和。

该演进路径在工程严谨度、测试便利性和实现复杂度之间保持平衡：起步简单、按需加码，避免一上来就把所有问题都交给事件、抽象接口或基础设施解决，也避免因缺失关键机制而在规模变大时返工。

# 17. 领域异常与 HTTP 边界映射

4.2 禁止 Service 依赖 `HTTPException`，12.2 中 Service 抛出的是 `InsufficientStockError` 这类**领域异常**。因此必须有一层专门负责把领域异常翻译为 HTTP 响应，否则分层不完整、异常会以 500 泄漏。

## 17.1 分层职责

```text
Service / 领域层
只抛领域异常，不知道 HTTP

边界映射层（异常处理器）
领域异常 → HTTP 状态码 + 错误响应体

Router / API 层
正常路径返回 Response Schema
```

## 17.2 领域异常层次

领域异常应有统一基类，便于分类映射：

```python
class DomainError(Exception):
    """所有领域异常的基类。"""


class NotFoundError(DomainError):
    ...


class ConflictError(DomainError):
    ...


class ValidationError(DomainError):
    ...


class InsufficientStockError(ConflictError):
    def __init__(self, product_id: UUID) -> None:
        self.product_id = product_id
        super().__init__(f"insufficient stock: {product_id}")
```

## 17.3 统一映射

在应用边界注册异常处理器，把领域异常类别映射到 HTTP 状态码，并输出统一的错误响应结构：

```python
_STATUS_MAP: dict[type[DomainError], int] = {
    NotFoundError: 404,
    ConflictError: 409,
    ValidationError: 422,
}


def register_domain_exception_handlers(app: FastAPI) -> None:
    @app.exception_handler(DomainError)
    async def handle_domain_error(
        request: Request,
        exc: DomainError,
    ) -> JSONResponse:
        status = _resolve_status(exc, _STATUS_MAP, default=400)
        return JSONResponse(
            status_code=status,
            content={
                "error": type(exc).__name__,
                "message": str(exc),
                "request_id": getattr(
                    request.state, "request_id", None
                ),
            },
        )
```

要求：

* 映射表集中管理，新增领域异常只需归类到已有基类；
* 未被显式映射的 `DomainError` 落到一个明确的默认状态码（如 400），而不是 500；
* 未知的非领域异常（真正的 bug）才返回 500，并记录完整堆栈；
* 错误响应结构统一，携带 `request_id` 便于追踪；
* Router 不写 `try/except` 逐个转换异常，交给统一处理器。

错误响应字段与错误码约定见 §6.3。

# 18. 测试策略

引入 UoW、Repository、DTO、事件后，必须配套测试方式，否则这些抽象只会增加编写测试的成本。

## 18.1 分层测试

* **领域 / Service 单元测试**：用 fake UoW + in-memory Repository 替换真实数据库，只验证业务规则与事务边界（是否 commit、异常是否回滚），不连库、快。
* **Repository / 集成测试**：连真实数据库（或等价容器），验证查询、关联加载、约束、迁移兼容。
* **API 测试**：走完整依赖链，验证请求/响应契约与领域异常到 HTTP 的映射（第 17 章）。

## 18.2 Fake UoW 与 Repository

面向 12.3 的 Protocol 编写内存实现，让 Service 测试无需数据库：

```python
class FakeOrderRepository:
    def __init__(self) -> None:
        self._items: dict[UUID, Order] = {}

    async def add(self, order: Order) -> None:
        self._items[order.id] = order

    async def get_required(self, order_id: UUID) -> Order:
        if order_id not in self._items:
            raise NotFoundError(order_id)
        return self._items[order_id]


class FakeUnitOfWork:
    def __init__(self) -> None:
        self.orders = FakeOrderRepository()
        self.products = FakeProductRepository()
        self.outbox = FakeOutboxRepository()
        self.committed = False

    async def __aenter__(self) -> "FakeUnitOfWork":
        return self

    async def __aexit__(self, *exc: object) -> None:
        return None

    async def commit(self) -> None:
        self.committed = True

    async def rollback(self) -> None:
        ...
```

要点：

* Service 注入 `uow_factory`，测试传入返回 FakeUoW 的工厂即可；
* 断言 `uow.committed` 验证提交时机；抛出领域异常时断言未提交，验证回滚语义；
* 断言 `uow.outbox` 中写入了预期事件，验证"业务写入与 Outbox 原子提交"。

## 18.3 集成测试的事务隔离

集成测试应保证用例之间互不污染。推荐**每个测试一个事务并在结束时回滚**（外层事务 + savepoint），而不是每次重建库：

* 每个测试开启一个外层事务，测试内的业务代码运行在其 savepoint 上；
* 测试结束统一回滚外层事务，数据不落库；
* 显著快于"每用例建表/清表"。

## 18.4 事件与幂等测试

* **进程内事件**：注册测试用 Handler，断言事件被发布、Handler 失败不影响主流程（对应 14.2 默认策略）。
* **Outbox**：断言业务提交后 Outbox 表存在对应消息；模拟"提交后崩溃"验证消息不丢（重启后 Worker 仍能发出）。
* **消费者幂等**：对同一 `message_id` 重复投递两次，断言业务只生效一次（对应 14.6）。

## 18.5 主动暴露懒加载与 N+1

呼应 6.4：测试环境应对不允许懒加载的关系配置 `lazy="raise"` 或查询级 `raiseload("*")`，让隐式懒加载在测试中直接失败而不是在生产序列化阶段爆 `MissingGreenlet`。必要时统计 SQL 执行次数，为关键查询设置 N+1 回归断言。
