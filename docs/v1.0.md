# FastAPI 项目架构与开发规范 v1.0

## 1. 参考项目与取舍

| 参考项目                        | 借鉴内容                                                            | 不直接照搬的部分                   |
| --------------------------- | --------------------------------------------------------------- | -------------------------- |
| FastAPI 官方文档                | `APIRouter` 拆分、依赖注入、路由级依赖、测试依赖覆盖                                | 官方示例偏教学，不包含完整业务分层          |
| Full Stack FastAPI Template | PostgreSQL、Alembic、Pytest、Docker Compose、GitHub Actions、认证和配置管理 | 模型和 CRUD 集中在少数文件，业务增长后容易膨胀 |
| fastapi-best-practices      | 按业务领域组织目录、异步规范、迁移规范、任务队列使用边界、Ruff                               | 部分建议带有作者团队偏好，需要按业务调整       |
| fastapi-best-architecture   | API、Schema、Service、CRUD、Model 三层划分，以及 Celery、监控等企业组件            | 组件较多，不适合项目早期全部引入           |
| Netflix Dispatch            | 按业务模块组织代码的思路                                                    | 项目已于 2025 年归档，只借鉴历史架构思想    |

FastAPI 官方明确支持通过多个 `APIRouter` 组织大型应用，并允许在路由级设置前缀、标签和依赖；测试时可以使用 `app.dependency_overrides` 替换数据库或外部服务依赖。

官方 Full Stack 模板提供 PostgreSQL、SQLModel、Docker Compose、Pytest、Alembic 和 GitHub Actions，但其后端仍以集中式 `models.py`、`crud.py` 和 API 路由为主，更适合作为工程化基础，而不是直接作为复杂领域架构。

社区的 fastapi-best-practices 推荐按领域组织模块，并覆盖异步接口、依赖复用、后台任务、数据库命名、Alembic 和 Ruff 等规范；它明确说明该模块化结构受到 Netflix Dispatch 启发。

## 2. 总体架构决策

推荐采用：

> **模块化单体 + 领域优先组织 + Service/Repository 分层 + 可选 Unit of Work**

先保持一个部署单元，不要在业务边界尚未稳定时拆微服务。

```text
HTTP Request
    ↓
API / Router
    ↓
Application Service
    ↓
Repository / External Client
    ↓
PostgreSQL / Redis / Third-party API
```

核心约束：

```text
API 可以依赖 Service
Service 可以依赖 Repository 接口
Repository 可以依赖 SQLAlchemy
下层不得反向依赖 API
业务模块不得直接访问其他模块的数据库表
```

### 3.1 为什么按业务模块组织

不推荐这种全局分类：

```text
routers/
services/
repositories/
models/
schemas/
```

当系统拥有几十个业务对象后，修改一个订单功能需要同时在五六个目录之间跳转。

推荐：

```text
modules/
├── orders/
├── payments/
└── identity/
```

一个业务功能的大部分代码位于同一个目录，更容易定位、测试和拆分。

## 4. 各层职责

### 4.1 Router

Router 只处理 HTTP 相关内容：

* 路径参数和查询参数；
* Pydantic 请求校验；
* 身份和权限依赖；
* 调用 Service；
* 设置状态码和响应模型。

禁止：

* 直接执行 SQL；
* 在路由中提交事务；
* 编写复杂业务判断；
* 直接调用多个 Repository 编排流程。

```python
@router.post("", response_model=OrderResponse, status_code=201)
async def create_order(
    command: OrderCreate,
    service: OrderServiceDep,
) -> OrderResponse:
    order = await service.create_order(command)
    return OrderResponse.model_validate(order)
```

### 4.2 Service

Service 负责：

* 业务规则；
* 用例流程；
* 权限之外的业务校验；
* 多个 Repository 的协作；
* 事务边界；
* 领域事件或异步任务发布。

```python
class OrderService:
    def __init__(
        self,
        order_repository: OrderRepository,
        product_repository: ProductRepository,
        unit_of_work: UnitOfWork,
    ) -> None:
        self._orders = order_repository
        self._products = product_repository
        self._uow = unit_of_work

    async def create_order(self, command: OrderCreate) -> Order:
        product = await self._products.get_required(command.product_id)

        if product.stock < command.quantity:
            raise InsufficientStockError(product.id)

        order = Order.create(
            product_id=product.id,
            quantity=command.quantity,
        )

        await self._orders.add(order)
        await self._uow.commit()
        return order
```

### 4.3 Repository

Repository 负责数据访问：

* 查询和持久化；
* 屏蔽 SQLAlchemy 查询细节；
* 返回明确的数据对象；
* 不包含业务规则。

Repository 默认不执行 `commit()`：

```python
class SqlAlchemyOrderRepository:
    def __init__(self, session: AsyncSession) -> None:
        self._session = session

    async def add(self, order: Order) -> None:
        self._session.add(order)

    async def get_by_id(self, order_id: UUID) -> Order | None:
        return await self._session.get(Order, order_id)
```

由 Service 或 Unit of Work 决定事务何时提交。

### 4.4 Schema 与 ORM Model

必须区分：

```text
Pydantic Schema：API 输入输出
SQLAlchemy Model：数据库持久化
```

禁止直接把 ORM Model 作为公开接口契约。数据库字段变化不应自动改变 API。

推荐命名：

```text
OrderCreate       创建请求
OrderUpdate       更新请求
OrderResponse     单项响应
OrderListResponse 列表响应
OrderFilter       查询条件
```

## 5. 模块边界规范

### 5.1 跨模块调用

允许：

```python
from app.modules.identity.service import IdentityService
```

避免：

```python
from app.modules.identity.repository import UserRepository
from app.modules.identity.models import User
```

订单模块需要用户信息时，应调用 Identity 模块公开的 Service 或查询接口，而不是直接查询用户表。

### 5.2 Shared 目录

`shared` 只能存放不包含具体业务语义的代码，例如：

* 分页结构；
* 通用时间处理；
* 基础类型；
* ID 生成器；
* 通用协议。

禁止把无法分类的代码全部放入 `shared/utils.py`。

### 5.3 是否需要纯领域模型

项目初期可以让 SQLAlchemy Model 同时承载部分领域行为：

```python
order.cancel()
order.confirm()
```

出现以下情况后，再将领域对象与 ORM Model 分离：

* 同一业务对象存在复杂状态机；
* 业务规则需要脱离数据库测试；
* 一个聚合包含多个实体和值对象；
* 持久化结构与业务模型差异明显。

不要为了形式上的 Clean Architecture 提前创建大量接口和映射器。

## 6. 开发规范

### 6.1 Python 与依赖管理

统一使用：

```text
pyproject.toml
uv
Ruff
Pyright 或 Mypy
Pytest
pre-commit
```

规范：

1. 锁定 Python 次版本。
2. 提交 `uv.lock`。
3. 生产依赖和开发依赖分组管理。
4. 禁止手工维护多份内容重复的 requirements 文件。
5. CI 必须从锁文件安装依赖。

### 6.2 代码风格

所有公开函数必须标注参数和返回类型：

```python
async def get_order(order_id: UUID) -> Order | None:
    ...
```

禁止：

```python
async def get_order(id):
    ...
```

其他要求：

* 使用绝对导入；
* 禁止 `from module import *`；
* 一个函数原则上只承担一个职责；
* 业务函数优先控制在约 40 行以内；
* 文件名、变量名使用 `snake_case`；
* 类名使用 `PascalCase`；
* 常量使用 `UPPER_SNAKE_CASE`；
* 不使用含糊名称，如 `data`、`obj`、`handle()`；
* 注释解释“为什么”，不要重复代码含义。

### 6.3 API 规范

统一版本前缀：

```text
/api/v1
```

路径使用资源名词：

```text
GET    /api/v1/orders
POST   /api/v1/orders
GET    /api/v1/orders/{order_id}
PATCH  /api/v1/orders/{order_id}
DELETE /api/v1/orders/{order_id}
```

动作型接口只用于无法自然表达为资源的操作：

```text
POST /api/v1/orders/{order_id}/cancel
POST /api/v1/orders/{order_id}/confirm
```

响应规则：

* 创建成功：`201`
* 删除成功且无响应体：`204`
* 参数错误：`422`
* 未认证：`401`
* 无权限：`403`
* 资源不存在：`404`
* 状态冲突：`409`

统一错误结构：

```json
{
  "error": {
    "code": "ORDER_INSUFFICIENT_STOCK",
    "message": "Insufficient product stock",
    "details": {
      "product_id": "..."
    },
    "request_id": "..."
  }
}
```

错误码必须稳定，前端不能依赖自然语言错误信息。

### 6.4 数据库规范

命名统一使用小写下划线：

```text
user
order
order_item
payment_transaction
```

数据库规则：

1. 所有外键建立索引。
2. 经常用于过滤、排序的字段按查询计划建立索引。
3. 唯一性约束必须由数据库保证。
4. 时间统一存储为 UTC。
5. 金额使用 `Decimal` 或数据库 `NUMERIC`，禁止使用浮点数。
6. 数据库 Session 按请求创建，禁止全局共享。
7. Repository 不自行提交事务。
8. 一个业务用例尽量对应一个事务。
9. 大批量处理禁止逐行查询，避免 N+1。
10. API 分页必须设置最大页大小。

### 6.5 Alembic 迁移规范

每次数据库结构变更必须提交迁移文件。

迁移文件命名：

```text
2026-07-02_add_order_status_index.py
```

要求：

* `upgrade()` 和 `downgrade()` 均可执行；
* 迁移内容必须静态、可审核；
* 禁止在应用启动时自动生成迁移；
* 禁止修改已经在生产执行过的迁移文件；
* 大表字段修改必须评估锁表时间；
* 数据迁移和结构迁移复杂时分开执行。

官方模板同样将 Alembic 迁移纳入版本控制，并要求模型变化后生成和执行迁移。

### 6.6 异步规范

使用 `async def` 的场景：

* 异步数据库驱动；
* HTTP 客户端；
* Redis；
* 消息队列；
* 文件或对象存储的异步 SDK。

禁止在事件循环中直接执行：

* `requests`；
* 同步数据库查询；
* 大文件压缩；
* 图片处理；
* 大规模数据计算；
* 长时间同步 SDK 调用。

短暂同步调用放入线程池。耗时、需要重试或不能丢失的任务进入 Celery、Dramatiq、Arq 等任务队列。

`BackgroundTasks` 只用于短小且允许丢失的非关键任务。社区最佳实践也指出，它运行在当前 Worker 进程中，没有持久化、重试和任务可见性，不应承担关键任务。

### 6.7 测试规范

测试分为四层：

```text
unit         Service 和纯业务逻辑
integration  Repository、数据库、Redis、外部客户端适配器
api          FastAPI 路由和依赖
e2e          完整关键业务流程
```

目录示例：

```text
tests/
├── unit/modules/orders/test_order_service.py
├── integration/modules/orders/test_order_repository.py
├── api/test_orders_api.py
└── e2e/test_order_payment_flow.py
```

规则：

* 单元测试不得连接真实数据库；
* 集成测试使用独立测试数据库；
* 每个测试自行创建数据；
* 禁止依赖测试执行顺序；
* 外部 API 必须 Mock 或使用本地 Stub；
* 时间、UUID 和随机数应可替换；
* 修复缺陷时先补充回归测试；
* CI 中禁止使用开发人员本地数据库。

FastAPI 官方提供 `dependency_overrides`，可在测试中将数据库、认证或外部服务依赖替换成测试实现。

### 6.8 日志与可观测性

日志使用结构化 JSON，至少包含：

```text
timestamp
level
service
environment
request_id
trace_id
user_id
method
path
status_code
duration_ms
```

要求：

* 每个请求生成或透传 `request_id`；
* 禁止记录密码、Token、Cookie 和银行卡信息；
* 异常日志保留堆栈；
* 业务异常记录为 `warning`，未知异常记录为 `error`；
* 提供 `/health/live` 和 `/health/ready`；
* 数据库和 Redis 状态只放在 readiness 检查；
* 指标至少包括请求量、延迟、错误率和数据库连接池状态。

### 6.9 安全规范

1. 密钥只从环境变量或 Secret Manager 读取。
2. `.env` 不进入版本库，仅提交 `.env.example`。
3. 密码使用 Argon2 或 bcrypt。
4. JWT 设置明确的签发方、接收方和过期时间。
5. CORS 使用允许列表，不允许生产环境通配符。
6. 所有列表接口设置分页上限。
7. 文件上传限制类型、大小和数量。
8. 管理接口单独设置权限依赖。
9. 关键操作记录审计日志。
10. CI 执行依赖漏洞和 Secret 扫描。

### 6.10 Git 与 Pull Request

分支命名：

```text
feat/order-cancellation
fix/payment-timeout
refactor/order-repository
```

Commit 示例：

```text
feat(order): support order cancellation
fix(payment): prevent duplicate callbacks
test(order): add cancellation conflict cases
```

Pull Request 必须包含：

* 变更目的；
* 设计说明；
* 数据库迁移影响；
* API 兼容性；
* 测试方式；
* 部署和回滚方式；
* 截图或请求示例，适用时提供。

禁止在同一个 PR 中同时进行大规模格式化和业务修改。

## 7. CI 质量门禁

### 7.1 提交前

```bash
ruff format --check .
ruff check .
pyright
pytest tests/unit
```

### 7.2 Pull Request

CI 至少执行：

```text
依赖锁文件检查
Ruff 格式检查
Ruff 静态检查
类型检查
单元测试
集成测试
Alembic 迁移测试
Docker 镜像构建
依赖漏洞扫描
Secret 扫描
```

官方 Full Stack 模板已将 Pytest、Docker 和 GitHub Actions 纳入默认工程流程。

### 7.3 发布前

发布前检查：

* 从空数据库执行全部迁移；
* 从上一生产版本执行增量迁移；
* 验证 OpenAPI 是否包含破坏性变化；
* 执行关键 E2E 流程；
* 验证健康检查；
* 准备回滚版本；
* 数据库破坏性变更采用先兼容、后清理的两阶段发布。

## 8. 组件引入时机

不要一开始安装全部基础设施。

| 需求             | 引入组件                          |
| -------------- | ----------------------------- |
| 普通 CRUD 和业务事务  | PostgreSQL、SQLAlchemy、Alembic |
| 热点读取和短期状态      | Redis                         |
| 可靠异步任务、重试、定时任务 | Celery、Dramatiq 或 Arq         |
| 全文检索           | OpenSearch 或 Elasticsearch    |
| 跨服务事件          | Kafka、RabbitMQ 或云消息服务         |
| 多实例追踪          | OpenTelemetry                 |
| 指标监控           | Prometheus、Grafana            |

社区的 fastapi-best-architecture 已集成 SQLAlchemy、Celery、Grafana、Docker 等企业组件，并采用 API、Schema、Service、CRUD、Model 的三层划分；这类方案适合作为功能清单参考，不代表每个项目都应全部启用。

## 9. 微服务拆分条件

满足以下多个条件时，再考虑拆分：

1. 模块拥有独立团队；
2. 发布周期明显不同；
3. 资源需求差异巨大；
4. 数据边界稳定；
5. 模块间调用关系简单；
6. 单体部署已经成为实际瓶颈；
7. 团队能够承担分布式事务、监控和运维成本。

拆分前应先保证模块间没有直接 Repository 和数据库表依赖。模块边界清晰后，才能把内部 Service 调用替换成 HTTP、RPC 或消息调用。

## 10. 最终建议

当前项目默认采用：

```text
模块化单体
+ 按领域组织目录
+ Router / Service / Repository
+ SQLAlchemy 2 Async
+ PostgreSQL
+ Alembic
+ Pydantic Settings
+ Pytest
+ Ruff
+ Pyright
+ uv
+ Docker
+ GitHub Actions
```

初期暂不引入：

```text
复杂 DDD 对象映射
事件溯源
CQRS
Kafka
Kubernetes
多个微服务
过度抽象的 Repository 接口
```

这套方案保留了 FastAPI 官方模板的简单性，同时补充了大型业务项目需要的模块边界、事务管理、测试分层和质量门禁。
